# 记忆注入产品文档

## 当前产品状态（2026-04-16）

### 已完成

- 首版记忆注入链路已经落地，能够基于固定触发点判断是否召回，并生成当前轮可用的注入块。
- 注入预算控制已经进入实现，包括优先级、条数和降级后的最小可用返回。
- 一轮结束后的写回检查已经落地，能够把有沉淀价值的内容整理成正式写回候选。
- 注入过程的运行轨迹已经可观测，可以追踪为什么触发、为什么没触发、为什么降级。

### 未完成

- 和宿主工具面联动的查询解释能力还没有补齐，当前缺 `memory_explain_hit`。
- `Codex` 侧的 `memory_search` 还不是正式查询工具，因此“手动查到并解释命中”的完整闭环还未完成。

### 当前结论

- 这个产品文档里的“自动恢复上下文”和“结束后写回检查”已经完成主体落地。
- 还未完成的是宿主工具辅助面，不影响自动注入主链路，但影响完整产品体验。

## 1. 文档目的

这份文档只讨论 `记忆注入` 这一侧的产品定义。

这里关心的不是记忆怎么存，而是：

- 什么时刻应该恢复记忆
- 恢复哪一类记忆
- 用什么方式把记忆放进 agent 当前上下文
- 在一轮交互结束后，哪些内容值得写回

一句话说，这个模块负责把“已经存在的记忆”在合适的时候真正变成“当前可用的上下文”。

这里把文档放在 `retrieval` 目录下，是因为它属于运行时召回链路的一部分，不代表它等同于底层查询模块。

## 2. 产品定义

记忆注入不是一个可选增强项，而是 agent 运行时的一层上下文调度机制。

它的职责是：

- 观察当前输入、会话状态和任务状态
- 判断当前是否应该触发记忆召回
- 组织一次面向当前问题的记忆恢复
- 控制注入规模，避免上下文被历史内容挤爆
- 在交互结束后，对本轮内容做写回检查

这个模块的核心价值不在“多查一点”，而在“在对的时刻，给对的记忆”。

## 3. 要解决的问题

### 3.1 模型不会稳定地主动查记忆

如果把记忆调用完全交给模型自己决定，会出现明显不稳定：

- 有时该查不查
- 有时查得过晚
- 有时查到一堆没用内容
- 有时重复问用户已经给过的信息

### 3.2 检索结果本身不等于当前上下文

即使记忆库已经返回了结果，也不代表这些内容适合直接进入 prompt（提示词）。

中间还需要有人做三件事：

- 判断哪些结果真的值得注入
- 压缩成当前轮可以接受的规模
- 按 agent 能理解和利用的形式摆进去

### 3.3 记忆会和当前任务抢上下文预算

如果没有注入层，历史内容很容易把当前任务本身挤掉。

所以这个模块天然要兼顾两件事：

- 尽量恢复关键上下文
- 尽量不给当前推理增加噪声

## 4. 产品目标

记忆注入模块的首版目标是：

- 让 agent 在关键时刻稳定恢复上下文
- 降低用户重复解释历史信息的次数
- 把记忆召回从“模型自己想不想查”变成“系统固定检查”
- 让注入内容最小但够用
- 让交互结束后的写回行为可控

对应的成功标准是：

- 该召回的时候能召回
- 不该召回的时候不过度打断
- 注入后的回复质量明显提升
- 上下文不会被注入内容压满

## 5. 核心职责

记忆注入模块负责下面五件事：

### 5.1 观察

读取当前轮的运行信息，包括：

- 当前用户输入
- 当前会话状态
- 当前任务状态
- 最近动作节点，比如刚开始规划、刚完成执行、刚切换任务

### 5.2 判断

根据固定策略判断当前是否需要召回记忆，以及需要哪一类记忆。

### 5.3 注入

把召回结果整理成 agent 当前轮可直接使用的上下文块。

### 5.4 控制

限制注入规模，处理冲突，避免把低价值记忆带进当前轮。

### 5.5 写回检查

在本轮结束后，判断是否产生了值得写回的新偏好、新事实、新任务状态或重要事件。

## 6. 不负责的事情

这个模块不负责：

- 底层存储和索引
- 记忆去重和持久化
- 直接维护原始聊天档案
- 让模型自由决定基础召回是否发生
- 用记忆代替 agent 做具体任务推理

## 7. 产品主链路

记忆注入模块的运行主链路如下：

1. 读取当前输入和当前运行状态
2. 检查是否命中触发点
3. 生成检索请求
4. 接收记忆包
5. 对记忆包做筛选、压缩和排序
6. 生成注入块
7. 注入到 agent 当前上下文
8. 等待 agent 完成回复
9. 对本轮结果做写回检查
10. 产出写回候选并提交给记忆库

这里最关键的控制点有两个：

- 第 2 步，避免不必要召回
- 第 5 步，避免把检索结果原样塞进去

## 8. 关键触发时机

首版至少支持下面这些触发时机：

### 8.1 会话开始

目的：

- 恢复稳定偏好
- 恢复活跃任务状态

### 8.2 任务开始

目的：

- 恢复该任务已有的目标、计划和历史决策

### 8.3 任务切换

目的：

- 回收旧任务上下文
- 切入新任务相关记忆

### 8.4 规划前

目的：

- 在 agent 开始列步骤前，先恢复约束、偏好、已决定方案和未完成事项

### 8.5 用户提到历史内容

典型表达包括：

- 中文：上次说过、之前定过、你还记得吗、我一般喜欢
- 英文：last time、previously、do you remember、I usually prefer

首版同时覆盖中英文关键词，完整列表见 `retrieval-runtime-implementation-spec.md`。

这类场景要强制触发相关记忆检查。

### 8.6 响应结束后

目的：

- 检查本轮是否形成了值得写回的新记忆

### 8.7 语义相似兜底

当上述六个触发点都没有命中时，系统在普通轮次里做一次轻量语义相似度检查。

做法是用当前用户输入的 embedding 和记忆库里活跃记录做一次快速比对。如果最高相似度超过阈值（首版建议 0.85），触发一次补充召回。

这个触发的目标是覆盖用户没有明确说"上次""之前"，但聊的话题和某条已有记忆高度相关的场景。

约束如下：

- 只在六个固定触发都没命中时才执行。
- 阈值首版设高，宁可漏掉也不要误触发。
- 不依赖模型判断，走向量索引。

## 9. 注入决策规则

记忆注入不是“只要查到就塞进去”。

首版先按下面规则判断：

### 9.1 优先级规则

当上下文预算有限时，优先级顺序是：

1. 事实与偏好
2. 当前任务状态
3. 情节记忆

### 9.2 最小注入规则

- 优先注入 `记忆包摘要`
- 只有在摘要不够时，才补少量支撑记录
- 单次默认只注入一个主记忆包

### 9.3 相关性规则

下列情况优先注入：

- 直接影响当前回答约束
- 直接影响当前任务续接
- 直接影响方案选择
- 能明显减少用户重复解释

下列情况默认不注入：

- 只是语义相似但当前用不上
- 和当前任务无直接关系
- 太旧且没有确认价值
- 和现有高可信记忆冲突但尚未确认

### 9.4 冲突处理规则

如果出现冲突记忆：

- 不直接当作事实注入
- 先降级成待确认信息
- 必要时让 agent 向用户确认

## 10. 注入内容格式

注入给 agent 的内容，首版固定包含三部分：

- `注入原因`
- `本次记忆摘要`
- `必要支撑记录`

格式原则如下：

- 摘要优先
- 分类清楚
- 来源可追溯
- 长度可控

注入时必须区分：

- 稳定事实与偏好
- 当前任务状态
- 历史事件参考

不能把三类内容混成一个大段落。

## 11. 上下文预算规则

记忆注入模块必须天然对上下文预算负责。

首版先固定这些约束：

- 单轮基础召回最多一次
- 单次注入默认最多携带 3 到 7 条记录
- 历史细节必须让位于当前任务输入
- 如果预算不足，先保留摘要，再删支撑记录
- 如果预算仍不足，先丢弃情节记忆

这个模块的目标不是“尽量多带”，而是“尽量带对”。

## 12. 写回检查规则

记忆注入模块在响应结束后，要执行一次固定检查。

### 12.1 适合写回的内容

- 新确认的用户偏好
- 稳定事实更新
- 任务目标变化
- 计划变化
- 明确承诺
- 对后续有价值的重要事件

### 12.2 不适合写回的内容

- 一次性闲聊
- 纯推理过程碎片
- 暂时性表达
- 未确认猜测
- 无法复用的局部噪声

### 12.3 写回输出

如果命中写回条件，模块输出的是 `写回候选`，而不是直接写库。

这层要做的是“提出建议”，不是“直接落库”。

## 13. 用户体验层面的表现

从用户视角，这个模块做得好，表现应该是：

- agent 更少重复追问已经说过的事
- agent 在任务续接时更自然
- agent 更容易维持已经约定好的偏好和限制
- agent 在记忆不确定时会主动确认，而不是假装知道

这个模块做得不好，通常会表现为：

- 乱提旧事
- 抢当前话题
- 把陈旧信息当成当前事实
- 回复里带入过多背景，显得拖沓

## 14. 对可观测平台的输出要求

虽然页面不属于这个模块本身，但这个模块必须向第三部分的可视化与观测平台输出结构化观测数据。

至少要输出下面这些内容：

- 本轮是否触发了召回
- 触发原因是什么
- 召回了哪类记忆
- 最终实际注入了什么
- 为什么没有注入
- 本轮是否产生写回候选
- 为什么产生或没有产生写回候选

这样做的目的，是让“为什么这轮带了这段记忆”变得可解释、可排查。

## 15. agent 集成方式

记忆系统和 agent 之间的集成，首版采用独立服务 + 薄 SDK 的方式。

### 15.1 独立服务

对 agent 暴露的是独立的 `retrieval-runtime` HTTP 服务，暴露 REST 接口。

主要接口包括：

- `/memory/query`：接收检索请求，返回记忆包
- `/memory/write-back`：接收写回候选，返回处理结果
- `/memory/observe`：输出可观测数据，供第三部分消费

这里的服务边界固定成下面这样：

- `retrieval-runtime` 负责对 agent 提供运行时查询、写回入口和本服务的可观测输出
- `storage` 是独立服务，负责写入处理和共享读模型发布
- `visualization` 是独立服务，负责页面和指标展示

接口协议选择 REST 的原因：架构独立性要求三部分只通过契约通信，REST 是最通用的契约形式，外部 agent 接入门槛最低。

### 15.2 薄 SDK

在 REST 接口之上提供一个客户端 SDK，封装 HTTP 调用和触发逻辑。

SDK 的职责只有两个：

- 把触发时机包装成明确的方法调用
- 封装底层 HTTP 请求细节

SDK 不做业务判断。agent 框架只需要在关键生命周期节点调用 SDK 的对应方法：

- `on_session_start`：会话开始
- `on_task_start`：任务开始
- `on_task_switch`：任务切换
- `on_before_plan`：规划前
- `on_before_response`：响应前（包含语义兜底检查）
- `on_after_response`：响应结束后（写回检查）

### 15.3 集成约束

- `retrieval-runtime` 不嵌入 agent 进程内部，必须作为独立服务部署。
- SDK 是可选的便利层，不是必须的。agent 也可以直接调用 REST 接口。
- 换 agent 框架时只需重新接 SDK 调用点，不需要改记忆系统本身。
- `storage` 或 `visualization` 未启动时，不影响 `retrieval-runtime` 自身启动和运行。
- 依赖失败只影响当前请求结果，不影响 `retrieval-runtime` 服务进程状态。

## 16. 代码层模块分离

虽然首版把检索和注入放在同一个 `retrieval-runtime` 部分里，但代码层面必须提前拆成两个内部模块：

- `retrieval`：负责接收检索请求、查库、排序、生成记忆包。
- `injection`：负责触发判断、注入决策、预算控制、写回检查。

两个模块之间通过内部接口通信，共用类型定义（`MemoryPacket`、`WriteBackCandidate` 等）放在独立的共享层。

检索侧读取的是 `storage` 发布的共享读模型，不是 `storage` 的私有写表。

这样做的目的是：首版部署简单（一个服务），后续如果要拆分，改的是部署配置，不是业务代码。

## 17. 首版范围

首版先做这些：

- 固定触发点
- 语义相似兜底触发
- 固定注入格式
- 固定预算规则
- 固定写回检查
- 固定冲突降级策略
- 提供注入侧基础可观测数据，供第三部分消费
- 独立服务 + 薄 SDK 集成方式
- 检索和注入代码层分离

首版先不做这些：

- 动态学习复杂触发策略
- 根据模型风格自动改变注入格式
- 多记忆包并行编排
- 多 agent 联合注入
- 长链路自主记忆调度

## 18. 关键指标

这部分后续至少要观察下面几类指标：

- 召回命中率
- 不必要召回率
- 用户重复说明率
- 注入后回复采纳率或完成率
- 单轮注入长度
- 写回候选通过率
- 触发后实际注入率

## 19. 一句话定义

记忆注入模块是 agent 外部的一层上下文恢复机制。

它负责在关键时刻，按规则把最小但足够有用的记忆恢复到当前轮里，并在交互结束后判断哪些新内容值得进入长期记忆。
