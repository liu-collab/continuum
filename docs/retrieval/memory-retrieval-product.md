# 记忆检索产品文档

## 当前产品状态（2026-04-16）

### 已完成

- 首版检索主链路已经可用，能够按触发上下文做结构化过滤、候选排序、结果裁剪和记忆包输出。
- 检索层已经按独立服务落地，并且直接读取共享读模型，不依赖 `storage` 私有写表。
- 查询阶段的低延迟约束已经进入实现，包括超时控制、取消和依赖降级。

### 未完成

- 面向宿主的手动工具查询面还没有完全交齐。
- `Codex` 侧还缺 `memory_explain_hit`，`memory_search` 也还没有变成正式工具查询。

### 当前结论

- 这个产品文档里的“运行时检索能力”已经进入可验收状态。
- 如果把“宿主工具化查询面”也算进产品范围，目前应标记为部分完成。

### 当前收口补充

当前阶段继续开发时，再补下面几条正式边界：

- 默认模式先收成 `single_local_user`
- 当前正式开放的作用范围是 `session`、`task`、`workspace`、`user`

## 1. 文档目的

这份文档只讨论 `记忆检索` 这一侧的产品定义。

这里关心的不是怎么把记忆写进去，而是：

- 当前轮应该怎么发起查询
- 查询请求应该带什么信息
- 怎么从存储层拿到最相关的结果
- 怎么把结果整理成上层可直接消费的记忆包

一句话说，这个模块负责把已经存好的记忆，在运行时快速、定向地取出来。

## 2. 产品定义

记忆检索模块是整个记忆系统的运行时查询层。

它的职责是：

- 对外提供 `查` 能力
- 接收来自注入模块的检索请求
- 按触发原因、任务范围和类型范围查询记忆
- 对结果做排序、裁剪和打包
- 返回统一格式的记忆包给上层

这个模块解决的是“当前这一轮怎么快速拿到最有用的记忆”。

在部署形态上，它属于独立的 `retrieval-runtime` 服务。
在数据访问上，它允许直接读取 `storage` 发布的共享数据库读模型，但不直接依赖 `storage` 的私有写表。

## 3. 要解决的问题

### 3.1 不能什么都查

如果每轮都宽泛搜索，很容易出现：

- 结果噪声大
- 相关性不稳定
- 响应变慢
- 上下文被无用历史挤占

### 3.2 检索必须快

检索在运行主链路里。
它一慢，整个 agent 回复就会被拖慢。

所以这部分必须天然优先考虑低延迟。

### 3.3 检索结果不能原样上抛

就算底层已经查到记录，也不能直接把生数据塞给上层。

中间还需要做：

- 结果裁剪
- 相关性排序
- 冲突处理
- 记忆包整理

## 4. 产品目标

记忆检索模块的首版目标是：

- 让检索在运行时保持低延迟
- 让检索结果和触发场景强相关
- 让返回结果格式统一稳定
- 让上层能直接消费，而不是自己拼底层记录

对应的成功标准是：

- 查询速度稳定
- 空检索和噪声检索可控
- 检索结果被上层采用的比例较高
- 记忆包能直接进入注入链路

## 5. 核心职责

记忆检索模块负责下面五件事：

### 5.1 接收检索请求

接收来自注入模块的结构化检索请求。

### 5.2 确定检索范围

根据触发原因、任务上下文和类型范围，确定本轮查询边界。

### 5.3 召回候选记录

从存储层准备好的结构化数据和索引里取回候选结果。

### 5.4 排序与裁剪

按相关性、重要度、可信度和时效性做结果裁剪。

### 5.5 生成记忆包

把最终结果整理成记忆包，再交给注入模块。

## 6. 不负责的事情

这个模块不负责：

- 记忆写入本身
- 去重、合并和归档本身
- 判断当前轮是否一定要召回
- 决定记忆如何进入 prompt（提示词）
- 承担可视化页面本身的产品职责

## 7. 检索请求要求

注入模块发起检索时，至少应提供：

- `trigger`（触发原因）
- `workspace_id`（当前工作区上下文）
- `user_id`（当前本地用户标识）
- `current_input`（当前输入摘要）
- `task_id`（当前任务标识）
- `session_id`（当前会话标识）
- `thread_id`（宿主线程标识，可选）
- `memory_mode`（`workspace_only` 或 `workspace_plus_global`）
- `requested_types`（希望召回的记忆类型）
- `result_limit`（结果数量上限）

约束如下：

- 不允许无限制召回
- 必须说明触发原因
- 必须有结果上限
- 外部正式契约不再直接传一个泛化的 `scope_limit`
- runtime 要根据 `memory_mode` 决定当前轮是否同时查询工作区记忆和全局记忆
- 查询接口必须默认走快路径

## 8. 记忆包要求

检索模块交付给注入模块的不是生记录，而是 `记忆包`。

字段完整定义见 `memory-module-contract.md` 第 4.2 节。首版记忆包至少包含：`packet_id`、`trigger`、`query_scope`、`records`、`packet_summary`、`injection_hint`、`ttl`。

约束如下：

- 记忆包必须围绕单一触发原因组织
- 必须先给摘要，再给记录
- 允许空结果，但必须说明原因
- 默认只对当前轮有效

## 9. 检索主链路

记忆检索模块处理召回请求时，主链路如下：

1. 接收检索请求
2. 读取触发原因
3. 确定记忆类型范围
4. 确定作用范围和任务范围
5. 召回候选记录
6. 做相关性排序和数量裁剪
7. 检查冲突和过时状态
8. 生成记忆包
9. 返回给注入模块

这条链路的目标不是查全，而是查准。

### 9.1 检索技术路径

首版采用结构化过滤 + 向量排序的混合检索。

具体分两步：

1. **硬过滤**：先用 `memory_type`、`scope`、`status`、`importance` 等结构化字段做精确筛选，缩小候选范围。这一步走 `storage` 发布的共享读模型上的 PostgreSQL 常规索引，速度快且确定性强。
2. **软排序**：在过滤后的候选集里，用 `summary` 的 embedding 向量和当前输入做相似度排序。这一步走 pgvector 索引。

这种路径的好处是：结构化过滤保证了类型和范围的精确性，向量排序保证了语义相关性。两步结合正好对应产品要求的"定向检索"。

## 10. 检索策略基线

首版先按触发场景提供固定检索策略。

### 10.1 会话开始

优先返回：

- 稳定偏好
- 当前活跃任务状态

### 10.2 任务开始

优先返回：

- 该任务的当前状态
- 最近相关的重要历史事件

### 10.3 任务切换

优先返回：

- 新任务已有目标
- 新任务最近决策
- 新任务未完成事项

### 10.4 规划前

优先返回：

- 约束
- 偏好
- 已有决策
- 未完成事项

### 10.5 用户提到历史内容

优先返回：

- 与被提及对象直接相关的偏好、事实或历史事件

### 10.6 语义兜底

当上述五类触发场景都没有命中时，在普通轮次里做一次轻量语义相似度检查。

做法是用当前用户输入的 embedding 和记忆库里的活跃记录做一次快速相似度比对。如果最高相似度超过阈值，触发一次补充召回。

约束如下：

- 只在前五类触发都没有命中时才执行。
- 阈值首版设高一点（建议 0.85），宁可漏掉也不要误触发。
- 走 pgvector 向量索引，不做全量扫描。
- 这个机制的目标是覆盖用户没有显式提到历史、但话题和某条记忆高度相关的场景。

## 11. 结果裁剪规则

为了保证检索结果可用，首版至少执行下面这些规则：

- 必须按上限裁剪结果
- 优先保留高重要度、高可信度、高相关度内容
- 摘要优先于细节
- 已明显过时的记录默认降权
- 有冲突但未解决的内容默认标记为待确认

## 12. 性能要求

这部分的产品要求先明确成下面几点：

- 查询速度优先级高于写入速度
- 查询必须走独立快路径
- 查询不能等待后台写入完成
- 查询结果允许来自索引、缓存或预整理结构

从产品视角，一句话就是：

查一定要快，而且不能被写拖慢。

### 12.1 性能指标基线

首版固定下面这组 P95 目标：

| 指标 | P95 目标 | 说明 |
|------|---------|------|
| 检索响应 | ≤ 200ms | 在 agent 主链路上，用户在等 |
| 记忆包组装 | ≤ 50ms | 检索结果到交付之间的整理耗时 |
| 空结果返回 | ≤ 100ms | 没有结果时应更快返回 |

查询链路默认读取 `storage` 发布的共享读模型。写入链路维护私有写模型，数据库层面应做读写分离或至少不共享锁。PostgreSQL 的 MVCC 机制天然支持这一点，但在高并发下仍需注意索引重建和 vacuum 对查询的影响。

验收时直接跑这组数字。

### 12.2 依赖不可用时的行为

如果 `storage` 未启动或暂时不可达：

- `retrieval-runtime` 进程仍然保持运行
- 查询请求必须在超时上限内快速返回
- 返回结果必须是显式的 `dependency_unavailable`（依赖不可用）或等价降级状态
- 不能把依赖失败伪装成普通空结果

## 13. 对可观测平台的输出要求

虽然页面不属于这个模块本身，但这个模块必须向第三部分的可视化与观测平台输出结构化观测数据。

至少要输出下面这些内容：

- 每次查询的触发原因
- 查询范围
- 查询耗时
- 返回结果数量
- 空结果原因
- 结果最终是否被上层采用

这个模块负责把数据准备好，不负责页面怎么画。

## 14. 首版范围

首版先做这些：

- 结构化检索请求
- 固定记忆包格式
- 按触发原因做定向检索
- 结构化过滤 + 向量排序的混合检索路径
- 语义相似兜底触发
- 结果排序和裁剪
- 查询快路径
- 向第三部分输出结构化观测数据

首版先不做这些：

- 复杂自学习检索策略
- 无监督动态重排
- 跨 agent 联合召回
- 可视化平台里的复杂自定义分析能力

## 15. 关键指标

这部分后续至少要观察下面几类指标：

- 空检索率
- 查询耗时
- 结果被注入率
- 触发后实际采用率
- 噪声检索率

## 16. 一句话定义

记忆检索模块负责在运行时，把已经存好的结构化记忆快速、定向地取出来，并整理成可直接进入注入链路的记忆包。
