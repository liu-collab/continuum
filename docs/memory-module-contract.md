# AI 记忆模块契约

## 1. 文档目的

这份文档用于定义核心运行模块之间的固定约束，并补充它们对观测平台的输出要求：

1. 记忆存储与检索模块
2. 记忆注入模块

目标不是把实现细节一次写完，而是先把两边的职责边界、数据接口和协作规则定清楚，避免后续一边做成数据库，一边做成临时拼接逻辑。

## 2. 模块划分

### 2.1 记忆存储与检索模块

这个模块负责：

- 对外提供 `查` 和 `写` 两类能力
- 在写入流程里完成标准化、去重、合并和状态更新
- 在查询流程里按规则检索记忆
- 把检索结果整理成统一格式返回

这个模块不负责：

- 决定什么时候一定要召回
- 决定当前回复里如何摆放记忆
- 直接改写 agent 的提示词
- 代替 agent 做推理判断

### 2.2 记忆注入模块

这个模块负责：

- 观察当前对话和运行状态
- 判断当前是否触发召回
- 生成检索请求
- 选择要注入的记忆包
- 把记忆包放到 agent 上下文中
- 在交互结束后决定是否发起写回

这个模块不负责：

- 直接管理底层存储
- 绕过记忆结构直接写原始内容
- 自己维护另一套独立记忆格式
- 让模型单独决定是否需要基础召回

## 3. 核心边界

两个模块之间的边界先固定成一句话：

记忆存储与检索模块负责“记忆内容本身”，
记忆注入模块负责“记忆在什么时刻、以什么形式进入当前上下文”。

进一步约束如下：

- 存储与检索模块输出的是结构化记忆结果，不直接输出大段原始历史。
- 注入模块消费的是结构化记忆结果，不直接查询底层表或索引。
- 注入模块可以决定本轮不用任何记忆，但不能跳过系统定义的触发检查。
- 存储与检索模块可以返回空结果，但不能替注入模块决定是否跳过召回流程。
- 存储与检索模块对外只暴露 `查询` 和 `写入` 两种动作，状态变更、去重、合并都属于写入链路内部处理。
- 后台治理动作可以存在，但默认不暴露给 agent 主链路。
- `retrieval-runtime` 可以读取由 `storage` 发布的共享读模型，但只能只读访问，不能依赖私有写表。

### 3.0 当前阶段收口边界

当前阶段先固定下面几条：

- 默认模式是 `single_local_user`（单本地用户）
- 当前正式开放的作用范围是 `session`、`task`、`workspace`、`user`
- `user` 表示全局记忆
- `workspace` 表示工作区记忆

这部分的统一方案见 `current-phase-closure-plan.md`。

### 3.0.1 当前边界说明

当前阶段不是多用户权限系统。

因此这里更重要的是使用边界，而不是鉴权头。

当前阶段统一约定：

- `user` 用来表示全局记忆空间
- `workspace` 用来表示工作区记忆空间
- 当前本地安装实例只有一个用户
- 工作区切换时，`workspace` 级记忆边界必须跟着切换
- runtime 必须支持“只用工作区记忆”或“工作区 + 全局记忆”
- 对 `scope=user` 的记录，保留来源工作区，但来源工作区不再等于可见性边界

运行模块和存储模块要遵守下面约束：

- 不把 `user` 再解释成多用户租户
- 不把 `workspace` 再解释成默认关闭的未来能力
- 页面、接口和写回策略都要明确区分全局记忆和工作区记忆

补一条硬约束：

- 运行时模块、存储模块、可观测平台都必须保持代码和运行边界独立，只能通过契约协作。

### 3.1 解耦约束

首版至少固定下面几条：

- 不允许直接跨模块引用内部实现。
- 不允许直接跨模块访问私有数据库结构。
- 不允许把一个共享大模块拆成几个目录后假装独立。
- 可观测平台只能消费正式输出的数据，不能读取运行时私有状态。
- 允许读取版本化共享读模型，这属于正式契约的一部分。

## 4. 统一对象定义

### 4.1 记忆记录

`记忆记录`（memory record）是记忆库里的最小可管理单位。

建议首版固定以下字段：

- `id`：记忆唯一标识
- `memory_type`：记忆类型，可取 `事实与偏好`、`任务状态`、`情节`
- `scope`：作用范围，可取 `全局级`、`工作区级`、`任务级`、`会话级`
- `summary`：记忆摘要，要求简洁、可直接复用
- `details`：结构化补充信息，不存整段原始对话
- `source`：来源信息，比如来自哪次会话、哪次任务、哪条事件
- `importance`：重要度，范围建议固定为 1 到 5
- `confidence`：可信度，范围建议固定为 0 到 1
- `created_at`：创建时间
- `updated_at`：更新时间
- `status`：状态，可取 `active`（有效）、`superseded`（已被更新）、`archived`（归档）
- `dedupe_key`：去重键，用于避免同一记忆重复写入

当前阶段状态语义进一步固定为：

- `active`：可以进入 runtime 默认召回
- `pending_confirmation`：待确认，不进入默认召回
- `superseded`：已被更新，不进入默认召回
- `archived`：归档，不进入默认召回
- `deleted`：软删除，只保留审计，不进入默认召回

约束如下：

- `summary` 必须能脱离原始上下文单独成立。
- `details` 必须是结构化字段，不允许直接塞完整聊天记录。
- `memory_type` 和 `scope` 必填。
- 新写入的记录必须携带 `source` 和时间信息。
- 同一条稳定事实被更新时，优先更新旧记录状态，而不是反复新增。
- 当前阶段正式开放的 `scope` 允许 `全局级`、`工作区级`、`任务级`、`会话级`。
- 其中 `user` 表示全局记忆，`workspace` 表示工作区记忆。
- 当 `scope=user` 时，记录里的 `workspace_id` 或 `source.origin_workspace_id` 表示来源工作区，用于解释来源，不用于限制跨工作区读取。

### 4.2 记忆包

`记忆包`（memory packet）是存储与检索模块返回给注入模块的交付单位。

建议首版固定以下字段：

- `packet_id`：记忆包唯一标识
- `trigger`：本次召回的触发原因
- `query_scope`：本次检索的范围说明
- `records`：本次返回的记忆记录列表
- `packet_summary`：面向注入的摘要
- `injection_hint`：建议注入方式，比如 `背景约束`、`当前任务续接`、`历史决策参考`
- `ttl`：本记忆包在当前流程中的有效时长

约束如下：

- 一个记忆包必须围绕一个明确触发原因组织。
- `packet_summary` 必须比 `records` 更短，可直接进入上下文。
- 记忆包默认是给当前轮使用的，不默认长期驻留。
- 记忆包里允许没有记录，但必须明确返回空结果原因。

### 4.3 写回候选

`写回候选`（write-back candidate）是注入模块在交互结束后提交给存储模块的待入库对象。

建议首版固定以下字段：

- `candidate_type`：候选类型
- `scope`：建议作用范围
- `summary`：候选摘要
- `details`：结构化内容
- `source`：来源上下文
- `importance`：建议重要度
- `confidence`：建议可信度
- `write_reason`：为什么要写回

约束如下：

- 写回候选不是最终记忆记录。
- 存储模块收到候选后，仍然可以做去重、合并、拒绝或更新。
- 写回候选必须说明写回理由，不能只给内容不给原因。
- 当前阶段写回候选的正式 `scope` 允许 `session`、`task`、`workspace`、`user`。

## 5. 模块接口约定

### 5.1 检索接口

记忆注入模块向存储与检索模块发起检索时，至少要提供：

- `trigger`：触发原因
- `current_input`：当前用户输入或当前任务事件摘要
- `task_id`：当前任务标识，没有则为空
- `session_id`：当前会话标识
- `requested_types`：期望检索的记忆类型列表
- `workspace_id`：当前工作区上下文
- `user_id`：当前本地用户标识
- `thread_id`：宿主线程标识，可选
- `result_limit`：返回条数上限
- `memory_mode`：当前运行模式，可取 `workspace_only` 或 `workspace_plus_global`

存储与检索模块返回：

- 一个记忆包
- 或者明确的空结果

约束如下：

- 检索接口不接受“把所有相关记忆都给我”这种无限请求。
- `result_limit` 必须有上限。
- 触发原因必须显式传入，不能省略。
- 查询链路必须优先保证低延迟，不能被后台写入流程阻塞。
- 下游依赖不可用时，必须返回显式错误或降级状态，不能无限等待，也不能伪装成正常成功。
- 当前阶段检索接口正式允许查询 `session`、`task`、`workspace`、`user` 四层范围。
- `memory_mode=workspace_only` 时默认不查询 `user`；`memory_mode=workspace_plus_global` 时允许同时查询 `workspace + user`。

### 5.2 注入接口

记忆注入模块把记忆交给 agent 时，至少要形成下面三部分：

- `injection_reason`：为什么此时要注入
- `memory_summary`：本次注入摘要
- `memory_records`：必要的支撑记录

约束如下：

- 注入默认优先用摘要，不直接堆完整记录。
- 只有在摘要不足时，才带少量支撑记录。
- 注入内容必须能区分“稳定事实”“当前任务状态”“历史事件”，不能混成一团。

### 5.3 写回接口

记忆注入模块在交互结束后，向存储与检索模块提交写回候选。

存储与检索模块收到后，需要做：

- 去重判断
- 冲突判断
- 合并或更新判断
- 最终写入或拒绝

约束如下：

- 写回接口只接收结构化候选，不接收整段 transcript（完整对话记录）。
- 存储模块首版正式返回异步接收结果 `jobs`，每项至少包含 `job_id`、`status=accepted_async`、`received_at`。
- 当前实现仍保留 `submitted_jobs` 兼容字段，方便旧调用方继续读取。
- 被拒绝时需要返回原因，方便后续调整写回策略。
- 写入接口允许先快速接收，再在后台完成去重、合并、索引和状态整理。
- 下游依赖不可用时，必须返回显式错误或降级状态，不能让调用方进程状态受影响。

### 5.4 治理接口

为了让错误记忆形成闭环，当前阶段补下面这组正式治理动作：

- `confirm`（确认）
- `edit`（修正）
- `invalidate`（标记失效）
- `archive`（归档）
- `delete`（软删除）
- `restore_version`（恢复历史版本）

约束如下：

- 每个治理动作都必须记录审计和操作原因
- 每个治理动作都必须刷新共享读模型
- runtime 默认只召回 `active` 记录
- 被治理后不再符合默认召回条件的记录，后续不能继续进入默认注入
- 当用户选择 `workspace_only` 模式时，runtime 默认不读取 `user`（全局记忆）

### 5.5 运行状态约定

三部分服务之间再补一条统一运行约定：

- 某个服务未启动，不影响其他服务启动。
- 某个服务不可达，不影响其他服务继续运行。
- 调用失败只影响当前请求，不影响调用方服务进程状态。

## 6. 触发规则

### 6.1 必选触发点

首版至少固定下面这些触发点：

- `session_start`（会话开始）
- `task_start`（任务开始）
- `task_switch`（任务切换）
- `before_plan`（规划前）
- `history_reference`（用户提到过去内容）
- `after_response`（响应结束后的写回检查）

### 6.2 触发规则约束

规则先定成下面这样：

- 会话开始时，默认检查 `事实与偏好` 和记忆中的活跃任务状态。
- 任务开始时，默认检查该任务下的 `任务状态` 和最近相关 `情节记忆`。
- 任务切换时，先回收旧任务上下文，再召回新任务相关记忆。
- 规划前，优先召回约束、历史决策和未完成事项。
- 用户显式提到”上次说过””之前定过””我偏好”这类内容时，强制触发相关检索。首版同时覆盖中文和英文关键词（如 `last time`、`previously`、`I prefer` 等），完整列表见 `retrieval-runtime-implementation-spec.md`。
- 每轮响应结束后，不一定写回，但一定执行一次写回检查。

## 7. 写回规则

### 7.1 应该写回的内容

首版建议只写回下面几类：

- 稳定的新偏好
- 新确认的事实
- 新形成的任务目标或任务状态变化
- 明确做出的承诺
- 会影响后续决策的重要历史事件

### 7.2 不应该写回的内容

首版明确不写回下面几类：

- 一次性闲聊
- 无法复用的短期噪声
- 纯过程性推理碎片
- 没有确认的猜测
- 与未来任务无关的重复表达

### 7.3 写回约束

- 同一轮交互中，写回数量必须受限，不能无限增长。
- 低重要度、低可信度内容默认不写回。
- 相同事实重复出现时，优先增强已有记忆，不新增重复记录。
- 任务结束后，任务状态类记忆应转为归档或摘要，而不是一直保持活跃。

## 8. 上下文预算约束

为了避免记忆把上下文挤爆，首版先定下面几个硬约束：

- 单次注入默认只允许一个主记忆包。
- 单次注入的记录条数必须有上限，建议先控制在 3 到 7 条。
- 摘要优先于原始细节。
- 当上下文预算不足时，优先保留 `事实与偏好`，再保留 `当前任务状态`，最后才是 `情节记忆`。

## 9. 性能约束

首版先固定下面几个性能方向：

- 查询速度优先级高于写入速度。
- 查询必须走独立快路径，不能等待写入完成。
- 写入允许异步化、后台化处理，只要最终能稳定入库。
- 如果可以同时做到查和写都快，优先采用，但不以牺牲查询稳定性为代价。
- 查询结果允许基于索引、缓存或预整理结果返回。

这部分最重要的一句话是：

查必须快，写可以稍慢，但写不能拖慢查。

## 10. 冲突与覆盖规则

当记忆之间发生冲突时，先按下面顺序处理：

1. 新近且高可信的记录优先
2. 用户显式确认过的记录优先
3. 任务级上下文优先于泛化的长期经验

约束如下：

- 冲突不能在注入时悄悄吞掉，必须保留冲突来源。
- 如果冲突无法自动解决，注入模块应把它降级成“待确认信息”，而不是直接当成事实。

## 11. 可观测性约束

为了后面能调策略，两个模块都必须向第三部分的可视化与观测平台输出最基本的可观测信息：

- 为什么触发了召回
- 召回了哪些类型的记忆
- 最终注入了什么
- 为什么写回或为什么没写回
- 某条候选为什么被合并或拒绝

首版不要求复杂监控，但这些基础日志必须保留。

### 11.1 可视化与观测平台的数据消费契约

第三部分 `visualization` 作为独立服务，它与前两部分的数据协作关系也属于正式契约的一部分。

`visualization` 允许消费的数据源：

- `storage` 发布的共享只读读模型（`memory_read_model_v1`）
- `storage` 的观测接口（`GET /v1/storage/observe/metrics`、`GET /v1/storage/observe/write-jobs`）
- `retrieval-runtime` 的观测接口（`GET /v1/runtime/observe/runs`、`GET /v1/runtime/observe/metrics`）

`visualization` 不允许做的事：

- 不允许读取 `storage_private` 的写表、版本表或冲突表
- 不允许读取 `runtime_private` 的运行时过程表
- 不允许直接改写记忆内容
- 不允许调用 `retrieval-runtime` 的主链路接口（`prepare-context`、`finalize-turn`）

`visualization` 的独立性约束：

- `storage` 或 `retrieval-runtime` 未启动时，`visualization` 仍然正常启动
- 对应数据区域显示"数据源不可用"，不因为上游不可达而导致自身不可运行
- 数据格式必须和上游在契约层约定好，不依赖对方内部实现细节
- 页面不允许暴露后端没有正式支持的筛选项
- 页面不允许展示未在正式契约里发布的字段
- 运行轨迹页当前正式只保留 `turn_id`、`session_id`、`trace_id`、`page`、`page_size`
- 运行轨迹详情应按 `turn`、`trigger`、`recall`、`injection`、`writeback` 五段解释

补充说明：

`visualization` 自身的后端进程不调用 `retrieval-runtime` 主链路接口。
如果用户通过 `visualization` 前端发起 agent 对话，主链路调用由 `memory-native-agent` 作为宿主发起，
`visualization` 只承担渲染职责。此种前后端跨服务聚合不视为耦合。

## 12. 第一版默认决策

为了避免首版过于发散，先固定下面这些默认决策：

- 记忆类型先只做三类：事实与偏好、任务状态、情节
- 召回主导权在外部系统，不在模型
- 检索结果必须先整理成记忆包，不能直接把底层结果塞进 prompt（提示词）
- 写回必须经过候选层，不能直接入库
- 每轮最多一次基础召回，额外深查以后再补
- 可视化与观测平台作为独立第三部分开发，不并入前两部分

## 13. 一句话约定

这两个模块之间最重要的约定是：

记忆库负责把记忆管好，
记忆注入模块负责把记忆在对的时刻、以对的规模、用对的形式放进 agent 当前上下文。
