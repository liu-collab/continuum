# 架构独立性约束

这份文档定义一个硬性前提：

三部分代码不能耦合在一起，必须保证每一部分都可以独立拆出、独立治理、独立运行。

同时补一条允许范围：

三部分必须是独立服务，但允许共享由 `storage`（记忆存储）正式发布的只读数据库读模型。
这个读模型属于契约层，不属于私有实现。

这里说的三部分，按业务代码和部署形态收成下面三个部分：

1. `storage`（记忆存储）
2. `retrieval-runtime`（运行时检索与注入）
3. `visualization`（可视化与观测）

另外允许存在一个独立的 `delivery / cli`（交付 / 命令层）模块。

说明：

- 这一层不算第四个业务服务
- 它只负责用户入口、安装、启动编排、状态探测
- 它不能把三个业务模块重新耦合成一个共享业务实现

说明：

从产品能力视角，`检索` 和 `注入` 是两件事。
但从代码和运行时边界看，首版可以把它们放进同一个 `retrieval-runtime`（运行时检索与注入）部分里，只要它对外边界稳定、内部职责清楚。

## 1. 核心原则

这条架构约束先固定成下面几句：

- 每个部分都必须可以单独部署。
- 每个部分都必须可以单独运行。
- 每个部分都必须可以单独治理。
- 每个部分之间只能通过明确契约通信，不能直接引用对方内部实现。
- 允许共享版本化的只读数据库读模型，但不允许共享写模型。
- 任意一个服务未启动、启动失败或暂时不可达，都不能把其他服务拖成不可运行状态。

这不是“尽量这样做”，而是产品架构上的硬约束。

## 2. 三部分各自负责什么

### 2.1 storage

这一部分负责：

- 记忆写入
- 记忆结构化保存
- 去重、合并、归档、失效
- 为查询提供稳定数据基础

这一部分不负责：

- 运行时是否触发召回
- 当前轮如何注入上下文
- 页面展示逻辑

### 2.2 retrieval-runtime

这一部分负责：

- 运行时发起检索
- 选择检索范围
- 生成记忆包
- 在当前轮完成记忆注入
- 响应结束后的写回候选生成

这一部分不负责：

- 直接管理底层存储实现
- 直接操作存储库表结构
- 承担页面展示逻辑

### 2.3 visualization

这一部分负责：

- 展示结构化记忆
- 展示运行轨迹
- 展示关键指标
- 提供统一观测和排查入口

这一部分不负责：

- 直接参与运行时召回决策
- 直接改写记忆内容
- 直接依赖前两部分的内部实现细节

## 3. 硬性解耦规则

首版至少遵守下面这些规则：

### 3.1 代码解耦

- 不允许一个部分直接引用另一个部分的内部类、内部函数或内部目录。
- 公共定义只能放在明确的契约层，不能把一方内部模型当成公共模型。
- 不允许为了省事，把三部分写成一个共享大模块后再用目录假装拆分。

### 3.2 数据解耦

- `retrieval-runtime` 可以直接读取 `storage` 发布的共享数据库读模型，但只能只读访问。
- `retrieval-runtime` 不能直接依赖 `storage` 的私有写表、私有索引或内部表结构。
- `visualization` 不能直接读取前两部分的私有运行内存。
- 页面展示所需数据，必须来自正式输出的事件、接口或共享读模型。

### 3.3 运行解耦

- 任意一个部分都应该能单独启动。
- 服务启动时不允许阻塞等待其他服务先启动。
- 远端依赖检查必须在启动后异步进行，不能作为进程启动前置条件。
- 任意一个部分不可用时，其他部分要么降级运行，要么返回显式错误，不允许整体绑死。
- 依赖不可用不能导致其他服务主动退出、卡死或进入重启循环。
- 本地开发和测试时，允许用桩服务或模拟接口替代其他部分。

### 3.3.1 健康状态分离

为了保证“服务没启动不影响别的服务状态”，首版固定三类状态：

- `liveness`（存活状态）：只反映本服务进程是否正常
- `readiness`（就绪状态）：只反映本服务是否能接收请求并给出正常或降级响应
- `dependencies`（依赖状态）：单独列出外部服务是否可用

约束如下：

- 外部依赖不可用时，不能直接把本服务 `liveness` 置为失败。
- 只要本服务还能接收请求并返回明确结果，`readiness` 就保持可用。
- 依赖异常必须显示在 `dependencies` 里，而不是混进本服务自身状态。

### 3.4 发布解耦

- 三部分必须可以独立发布。
- 三部分必须可以独立回滚。
- 三部分必须有自己的配置和版本边界。

### 3.5 治理解耦

- `storage` 的治理动作归 `storage` 自己负责。
- `retrieval-runtime` 的触发规则和注入策略归 `retrieval-runtime` 自己负责。
- `visualization` 的页面、指标和查看体验归 `visualization` 自己负责。

不能把治理职责混写到一起。

## 4. 通信方式约束

三部分之间只允许通过下面几类正式方式通信：

- 版本化接口
- 版本化事件
- 明确的结构化对象
- 版本化共享读模型（只读）

不允许：

- 直接跨模块调用内部函数
- 直接读写对方私有数据库
- 通过临时脚本共享未定义的数据结构

关于共享读模型，再补三条：

- 共享读模型由 `storage` 负责定义、发布和版本管理。
- `retrieval-runtime` 和 `visualization` 只能读取，不能反向改写。
- 共享读模型变化时，必须通过版本升级，而不是静默改表。

关于运行时调用，再补两条：

- 所有跨服务调用都必须有超时上限，不能无限等待。
- 所有跨服务失败都必须返回显式错误类型，不能伪装成成功。

### 4.2 跨服务前端嵌入

允许一个服务的前端页面调用另一个服务的公共 HTTP/WebSocket 接口，
以实现用户可见的聚合体验。前提是：

- 被调用方仍然是独立服务、独立部署、独立治理
- 调用方的后端进程不承接被调用方的业务逻辑
- 调用方只通过正式契约通信，不引用被调用方的内部实现

首版已落地场景：

`visualization` 前端 `/agent/*` 路由直接调用 `memory-native-agent` 的 HTTP/WS 接口。

### 4.1 外部共享依赖约定

三部分服务之外，还存在一些共享的外部依赖。首版最主要的共享外部依赖是 **向量 embedding 服务**。

embedding 服务的职责归属约定如下：

- embedding 服务是三部分之外的独立外部依赖，不归属于任何一个服务。
- `storage` 在写入链路中使用 embedding 服务为 `summary` 生成 `summary_embedding`，写入共享读模型。
- `retrieval-runtime` 在查询链路中使用 embedding 服务为当前用户输入生成查询向量，用于语义排序和语义兜底触发。
- 两个服务各自独立配置 embedding 服务连接，不通过对方中转。
- embedding 服务不可用时，`storage` 侧正式记录仍然入库，但 `summary_embedding` 字段暂时为空，后续补刷。
- embedding 服务不可用时，`retrieval-runtime` 侧退回纯结构化过滤，不做语义排序，返回显式降级标记。
- `visualization` 不直接调用 embedding 服务。

## 5. 降级与容错

为了保证“独立运行”不是一句空话，首版先明确下面几条：

- `storage` 暂时不可用时，`retrieval-runtime` 仍保持运行，但查询和写回要快速返回 `dependency_unavailable`（依赖不可用）或等价的显式降级结果。
- `visualization` 不可用时，前两部分仍应正常运行。
- `retrieval-runtime` 不可用时，`storage` 仍可继续接收治理动作和后台写入。
- `storage` 或 `retrieval-runtime` 不可用时，`visualization` 仍应能启动并显示“数据源不可用”，而不是自身不可运行。

## 6. 验收要求

这条架构约束必须进入验收。

首版至少验证下面几件事：

- 三部分能否分别单独启动
- 三部分能否分别单独部署
- 三部分是否只通过契约通信
- 去掉某一部分后，其他部分是否仍能按预期降级或独立运行
- 任意一个服务未启动时，其他服务是否仍能保持自身 `liveness` 和 `readiness`

## 7. 首版一句话约束

首版架构必须满足：

三部分不是一个系统里的三个目录，而是三个边界清楚、可独立运行、可独立治理、可独立发布的部分。
