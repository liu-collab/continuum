# storage 实质改进提示词

## 1. 这份提示词给谁

这份提示词给负责继续收口 `services/storage` 的开发 agent。

当前这层已经不是“缺主链路”，也不是“只有表结构和接口骨架”。

现在的状态是：

- 服务能启动
- `check`、`build`、`test` 已通过
- 写回入口、异步处理、共享读模型、治理刷新、embedding 刷新、健康接口都已经有实现

但这层还没有达到“正式契约完全一致、冲突治理完全闭环、真实数据库验收充分”的状态。

问题已经不在基础能力，而在剩余几个会影响正式收口的硬缺口。

你的任务不是重写 `storage`，而是把这些剩余改进项收口。

## 3. 当前还剩哪些实质改进项

### 3.1 `records` 查询接口的正式契约还没有收口

当前最明确的正式缺口，是 `docs/api-contract.md` 和真实实现之间还存在契约漂移：

- 契约写 `workspace_id` 必填，代码里仍是可选
- 契约写正式分页，代码里仍只有 `limit`

这会直接影响后续联调和页面侧实现，因为上游无法判断该以哪套规则为准。

### 3.2 写回接收接口的返回体还没有统一成一套正式口径

当前单条写回和批量写回虽然都能用，但返回结构还没有收成一套稳定契约。

这会带来一个实际问题：

- 上游能接通
- 但很难把返回结果当成正式、稳定、可长期依赖的接口契约

### 3.3 冲突治理还缺“新旧双方都可治理”的正式结构

当前实现已经能识别冲突、挂起旧记录、打开冲突单。

但在不能自动解决的分支里，新的冲突候选还没有成为第二个正式可对比对象。

这意味着：

- 可以发现冲突
- 但冲突治理还不是完整的双边治理结构

### 3.4 还缺更强的真实数据库验收证据

当前 `check`、`build`、`test` 已通过，但真实 PostgreSQL 场景还有跳过项。

这不是说当前主链路不可用，而是说：

- 代码证据已经够说明“主路径成立”
- 但还不够说明“正式收口时数据库级闭环没有遗漏”

## 4. 你的目标

这次开发目标只有一个：

把 `storage` 从“主链路已经能跑”补到“契约一致、冲突治理更完整、数据库级验收证据完整”。

## 5. 你必须补齐的内容

### 5.1 统一 `records` 查询接口的正式口径

你要把代码和契约收成同一套规则。

至少要明确：

- `workspace_id` 是正式必填，还是正式可选
- 列表查询是正式分页，还是正式 `limit` 查询

### 5.2 统一写回接收接口的返回体口径

你要把单条和批量写回的返回结构收成一套稳定契约。

要求：

- `job_id` 这类任务映射关系不能丢
- 不要再保留两种风格明显不同的正式返回体

### 5.3 补完整冲突双边治理结构

你要补的是冲突治理结构，而不是冲突识别逻辑。

至少要做到：

- 冲突新候选和旧记录都能被治理层明确识别
- 冲突单里能准确表达“谁和谁冲突”
- 后续确认、恢复、删除、归档的行为能作用在明确对象上

### 5.4 补更强的数据库级验收测试

你要补的不是普通单测，而是更接近正式验收的证据。

至少覆盖：

- 正式接口契约和真实数据库路径一致
- 冲突治理在真实数据库路径上成立
- 共享读模型发布在真实数据库路径上成立

## 6. 这次开发不要做什么

这次不要偏题。

不要把精力花在下面这些方向：

- 不要重写领域规则
- 不要扩展新的记忆类型
- 不要顺手改 `retrieval-runtime` 逻辑
- 不要只改 README 不改真实行为
- 不要只补内存仓储测试，绕开真实数据库闭环

## 7. 完成标准

完成后，至少要满足下面这些条件：

### 7.1 契约层

- `records` 查询接口和正式契约一致
- 写回接收接口返回体和正式契约一致

### 7.2 治理层

- 冲突双方都能进入正式治理闭环
- 冲突单不再只有单边信息

### 7.3 测试层

- 新增测试覆盖这些改进项
- `npm run check`
- `npm run build`
- `npm test`

都通过

4. 新增了哪些数据库级或契约级测试
5. 还有没有剩余风险
