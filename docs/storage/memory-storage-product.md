# 记忆存储产品文档

## 1. 文档目的

这份文档只讨论 `记忆存储` 这一侧的产品定义。

这里关心的不是当前轮怎么查，而是：

- 什么内容值得入库
- 入库后怎么结构化保存
- 怎么做去重、合并和状态管理
- 怎么为后续快速检索准备好稳定的数据基础

一句话说，这个模块负责把零散交互沉淀成长期可管理、可复用的结构化记忆。

## 2. 产品定义

记忆存储模块是整个记忆系统的内容沉淀层。

它的职责是：

- 对外提供 `写` 能力
- 接收写回候选
- 把候选整理成标准记忆记录
- 管理记忆状态和生命周期
- 为检索模块提供稳定、结构化、可索引的数据基础

这个模块解决的是“记忆如何被稳定保存和持续管理”。

在部署形态上，它是独立服务。
在数据协作上，它负责发布一个只读共享读模型，供 `retrieval-runtime` 和 `visualization` 消费。

### 当前收口补充

当前阶段已经收成下面这些正式边界：

- 默认模式先收成 `single_local_user`
- 当前正式开放的作用范围是 `session`、`task`、`workspace`、`user`
- `workspace` 表示工作区记忆，`user` 表示全局记忆
- 存储层已经正式支持 `workspace` 级写入和治理
- 治理能力已经落地 `确认、修正、失效、归档、删除、恢复版本`

当前还没有完全收口的点：

- `GET /v1/storage/records` 的正式契约和代码行为还没有完全一致
- `POST /v1/storage/write-back-candidates` 的返回体口径和正式契约还没有完全一致
- 冲突治理目前是最小可用闭环，还不是完整的双边对比治理闭环

具体方案见 `../current-phase-closure-plan.md`。

## 3. 要解决的问题

### 3.1 原始对话不能直接入库

如果把完整聊天内容直接塞进记忆库，会很快出现：

- 信息膨胀
- 噪声堆积
- 稳定事实和临时表达混在一起
- 后续很难维护，也很难稳定检索

### 3.2 同一事实会反复出现

用户偏好、稳定事实和任务状态会多次被提到。
如果没有去重和合并，记忆库很快会变成重复堆积。

### 3.3 记忆需要生命周期

不是所有记忆都应该一直活跃：

- 有些事实会被更新
- 有些任务状态会结束
- 有些低价值记忆会逐渐失效

所以存储不是“写进去就不管了”，而是持续治理。

## 4. 产品目标

记忆存储模块的首版目标是：

- 建立结构化的记忆记录格式
- 保证写入内容干净、可复用、可追溯
- 控制重复、冲突和过时内容
- 让存储结果天然适合后续快速检索

对应的成功标准是：

- 记忆库里不是原始对话堆积
- 同一事实不会无限重复
- 任务状态有明确状态流转
- 存储结果能稳定支撑检索快路径

## 5. 核心职责

记忆存储模块负责下面六件事：

### 5.1 接收写回候选

接收上游提交的结构化 `写回候选`。

### 5.2 标准化

把候选整理成统一的记忆记录格式。

### 5.3 去重与合并

识别重复事实、相近偏好和同一任务状态更新。

### 5.4 状态管理

维护记忆记录的状态，比如：

- 活跃
- 已更新
- 已归档

### 5.5 生命周期治理

在后台处理：

- 确认
- 修正
- 失效
- 归档
- 删除
- 恢复版本
- 冲突标记和治理刷新

### 5.6 对检索提供数据基础

为检索模块提供稳定记录、状态信息和可快速访问的索引结果。

## 6. 不负责的事情

这个模块不负责：

- 判断当前轮是否应该召回
- 决定记忆如何进入 prompt（提示词）
- 代替注入模块做运行时上下文选择
- 承担可视化页面本身的产品职责

## 7. 记忆记录结构

记忆记录是存储模块里的最小管理单位。

字段完整定义见 `memory-module-contract.md` 第 4.1 节。首版至少包含：`id`、`memory_type`、`scope`、`summary`、`details`、`source`、`importance`、`confidence`、`created_at`、`updated_at`、`status`、`dedupe_key`。

约束如下：

- `summary` 必须短且可直接复用
- `details` 必须结构化
- 不允许直接把完整对话原文当作 `details`
- `memory_type` 和 `scope` 必填
- 所有记录都必须可追溯来源
- `scope=user` 的记录允许跨工作区复用，但要保留来源工作区用于解释
- `scope=workspace` 的记录只在对应工作区内复用

### 7.1 可信度打分规则

`confidence` 取值 0 到 1，按记忆来源确定：

| 来源 | 默认可信度 |
|------|-----------|
| 用户明确表达 | 0.9 |
| 用户确认过的系统推断 | 0.8 |
| agent 完成的任务状态更新 | 0.7 |
| 系统从对话中提取但未经确认 | 0.5 |

### 7.2 重要度打分规则

`importance` 取值 1 到 5，按记忆类型和影响面确定：

| 条件 | 默认重要度 |
|------|-----------|
| 用户级稳定偏好或约束 | 5 |
| 任务目标变化、明确承诺 | 4 |
| 任务状态更新 | 3 |
| 一般历史事件 | 2 |
| 临时性上下文补充 | 1 |

### 7.3 打分约束

- 写入时必须按上述规则赋初始分，不允许留空。
- 后续如果用户确认或否定了某条记忆，可信度可以上调或下调。
- 排序和裁剪依赖这两个字段，打分一致性直接影响检索质量。
- 首版不引入模型打分。如果后续规则不够用，在这套基线上叠加统计调整。

## 8. 写入主链路

记忆存储模块处理写回时，主链路如下：

1. 接收写回候选
2. 校验字段完整性
3. 判断是否属于可写入范围
4. 识别类型和作用范围
5. 计算去重键或匹配已有记录
6. 做冲突、合并或更新判断
7. 完成持久化
8. 更新状态和索引
9. 返回处理结果

这条链路的关键点不是“尽量收”，而是“只收应该留下的东西”。

## 9. 写入规则基线

首版建议只接收下面几类写回：

- 新确认的偏好
- 新确认的事实
- 任务目标变化
- 任务状态变化
- 明确承诺
- 对未来有影响的重要事件

首版默认拒绝下面几类写回：

- 一次性闲聊
- 纯推理草稿
- 未确认猜测
- 短期噪声
- 重复表达

## 10. 治理规则

为了让记忆库长期可用，首版先预留下面这些治理规则：

### 10.1 去重规则

- 同一事实不重复新增
- 相近偏好优先合并
- 同一任务状态以最新记录为主

### 10.2 过时处理规则

- 已完成任务的过程状态不能长期保持活跃
- 被新信息覆盖的旧事实应标记为已更新
- 长期未使用且价值低的记录可归档

### 10.3 冲突处理规则

- 冲突来源必须保留
- 无法自动解决的冲突要标记出来
- 冲突记录不能静默覆盖

## 11. 性能要求

这部分的产品要求先明确成下面几点：

- 写入可以比查询慢。
- 写入允许异步化和后台化。
- 只要不影响查询链路，去重、合并、索引刷新可以放到后台。
- 如果可以做到写也快，那更好，但不是首版强要求。

从产品视角，一句话就是：

写可以慢一点，但不能把库写坏，也不能拖慢查。

### 11.1 技术选型基线

首版存储层使用 PostgreSQL + pgvector。

选择理由：

- 记忆记录结构规整，字段类型固定，天然适合关系型存储。
- 检索链路需要先按 `memory_type`、`scope`、`status` 做硬过滤，再按语义相关性排序。PostgreSQL 负责结构化过滤，pgvector 负责向量排序，两步在同一个库里完成。
- 产品要求的数据隔离（用户隔离、工作区隔离）在关系型数据库里用行级策略实现最直接。
- PostgreSQL 的 MVCC 机制天然支持读写不互锁，满足"写不能拖慢查"的要求。

不选纯向量库的原因：产品检索路径是"先过滤再排序"，不是"在全量记忆里做语义搜索"。结构化过滤 + 向量排序的混合路径比纯向量库更匹配。

### 11.2 共享读模型

虽然 `storage` 是独立服务，但允许它发布共享数据库读模型。

首版建议做法：

- `storage` 维护私有写模型
- 同时在同一个 PostgreSQL 实例或只读副本里发布版本化共享读模型
- `retrieval-runtime` 和 `visualization` 只读取共享读模型，不读取私有写表

约束如下：

- 共享读模型是正式契约的一部分
- 共享读模型变更必须有版本管理
- 写模型内部改动不能直接泄漏给外部服务

### 11.3 异步写入

写回候选通过 `PostgreSQL job table + polling worker`（数据库任务表与轮询 worker）异步投递到存储模块。

- 存储模块收到候选后，先快速确认接收，再在后台完成标准化、去重、合并和索引更新。
- 这条异步链路保证写入处理不会阻塞查询快路径。

### 11.4 性能指标基线

首版固定下面这组 P95 目标：

| 指标 | P95 目标 | 说明 |
|------|---------|------|
| 写回确认 | ≤ 500ms | 只确认收到，实际处理异步 |
| 写入完成（含去重合并） | ≤ 3s | 后台处理，用户无感 |
| 写入耗时 | 不阻塞查询链路 | 读写必须在数据库层面不共享锁 |

验收时直接跑这组数字。

## 12. 对可观测平台的输出要求

虽然页面不属于这个模块本身，但这个模块必须向第三部分的可视化与观测平台输出结构化观测数据。

至少包括：

- 记忆记录结构化视图
- 记录状态变化
- 写入耗时
- 接受、合并、拒绝结果
- 对应原因

## 13. 首版范围

首版先做这些：

- 标准记忆记录结构
- 基于规则的可信度和重要度打分
- 写回候选处理
- 去重与状态管理
- 基础生命周期治理
- 为检索快路径准备结构化数据
- 向第三部分输出结构化观测数据
- PostgreSQL + pgvector 存储
- 异步写入链路（`PostgreSQL job table + polling worker`）

首版先不做这些：

- 复杂情绪记忆建模
- 全自动长期重写
- 大规模图谱治理
- 完整人工运营后台

## 14. 关键指标

这部分后续至少要观察下面几类指标：

- 记忆写入接受率
- 记忆合并率
- 重复记录率
- 写入耗时
- 状态更新成功率

## 15. 一句话定义

记忆存储模块负责把可保留的信息写成结构化记忆，并持续管理这些记忆的状态、去重和生命周期。
