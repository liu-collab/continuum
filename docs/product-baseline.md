# AI 记忆产品基线

## 1. 产品目标

我们要做的是一个 AI 记忆产品，核心目标只有一个：

让记忆成为 agent 在关键时刻可靠恢复上下文的机制，而不是一个被动存在、要等模型自己想起来才去搜索的数据库。

落到实际设计上，就是：

- 系统自己决定什么时候需要召回记忆。
- 系统自己决定当前应该注入哪一类记忆。
- 模型可以使用记忆，但不单独承担触发记忆的责任。

## 2. 核心判断

现在很多记忆方案都围绕 `storage + retrieval`（存储 + 检索）来做。
这一步是必要的，但不够。

真正的问题不只是“记忆能不能被检索到”。
更关键的是“记忆能不能在对的时刻被恢复进当前上下文”。

所以这个产品分成三部分：

1. 记忆检索与写入
2. 面向 agent 的记忆注入
3. 可视化与观测平台

第一部分解决“记忆存在且可取用”。
第二部分解决“记忆真正被用上”。
第三部分解决“记忆系统是否可见、可查、可解释”。

## 3. 我们要做什么

我们要做的是一个面向 agent 的三部分记忆产品。

### 3.1 记忆检索与写入层

这一层负责：

- 接收和保存记忆
- 对记忆做结构化和索引
- 按类型和上下文检索相关记忆
- 返回结构化的记忆包，而不是原始记录堆叠

这一层回答的问题是：

- 什么内容应该写进记忆？
- 记忆应该怎么组织？
- 记忆应该怎么检索？

### 3.2 记忆注入层

这一层负责：

- 判断当前时刻是否需要召回记忆
- 选择当前应该注入的记忆包
- 在推理前或推理过程中，把记忆注入 agent 上下文
- 在交互结束后，判断这次内容是否要写回记忆

这一层回答的问题是：

- 什么时候应该召回记忆？
- 现在应该注入哪一部分记忆？
- 什么时候应该保存新的记忆？

### 3.3 可视化与观测平台

这一部分负责：

- 展示结构化记忆内容
- 展示关键运行指标
- 展示召回、注入、写回的过程结果
- 为调试、排查和运营提供统一观察入口

这一部分回答的问题是：

- 当前记忆库里到底有什么？
- 这一轮为什么查了、为什么注入了、为什么写回了？
- 当前系统运行得稳不稳、快不快、准不准？

## 4. 产品范围

### 4.1 首版范围内

第一版先聚焦下面这些内容：

- 一个结构化的记忆模型
- 记忆写入规则
- 记忆检索规则
- 一个独立于模型之外的召回触发机制
- 一条面向 agent 的记忆注入流程
- 一条交互结束后的记忆写回流程
- 一个独立的可视化与观测平台

### 4.2 第一版暂不处理

第一版不打算一次性把所有问题都做完。
下面这些内容先不放进首版范围：

- 完整的长期人格化建模
- 完全自主的记忆演化
- 复杂的情绪记忆建模
- 多 agent 之间的共享记忆网络
- 没有明确策略约束的自优化检索

## 5. 记忆模型

记忆层不能只是一个没有区分的大池子。
至少要先分成下面几类：

### 5.1 事实与偏好记忆

用于保存相对稳定的信息，比如：

- 身份事实
- 用户偏好
- 固定规则
- 长期约束或长期指令

### 5.2 任务状态记忆

用于保存正在进行中的工作上下文，比如：

- 当前目标
- 当前计划
- 未完成事项
- 已经做出的关键决策

### 5.3 情节记忆

用于保存过去发生过的重要事件和交互经历，比如：

- 重要对话
- 已经做出的承诺
- 之前解决问题的路径
- 明显的方向调整

## 6. 产品原则

系统应当遵循下面几个原则：

### 6.1 记忆不是原始档案

记忆不是完整对话逐字存档。
它应该是压缩过、结构化、可直接使用的内容。

### 6.2 召回应由系统策略触发

记忆召回不能完全依赖模型是否记得调用工具。
外部系统要在关键时刻主动触发召回。

### 6.3 注入要尽量少，但必须够用

系统应该注入最小但足够有用的记忆包，而不是一次塞进大量历史内容。

### 6.4 写回必须有选择

不是每一次交互都应该变成记忆。
只有稳定、有用、重要的信息才应该被保存。

### 6.5 三部分必须架构解耦

`storage`（记忆存储）、`retrieval-runtime`（运行时检索与注入）、`visualization`（可视化与观测）必须是三个可独立拆出、独立治理、独立运行的部分。

它们之间只能通过明确契约协作，不能在代码和数据层直接耦合。

允许的例外是：

可以共享由 `storage` 正式发布的只读数据库读模型，但不能共享写模型和私有表结构。

## 7. 核心机制

这个产品需要实现一条固定的记忆循环：

1. 观察当前输入和运行状态
2. 判断是否需要召回记忆
3. 从记忆库中取出相关记忆包
4. 把记忆注入 agent 当前上下文
5. 由 agent 完成推理和响应
6. 判断这次交互中是否有新记忆需要写回

这条循环就是产品主链路。

## 8. 关键触发时机

记忆注入层至少要支持下面这些触发时机：

- 会话开始
- 任务开始
- 任务切换
- 执行前规划
- 用户提到过去的决策、偏好或历史内容
- 响应结束后的写回检查

这些触发点应该是明确的产品行为，不应该只是模型“也许会想起来”。

## 9. 这个产品能带来什么

如果这套基线做对了，产品应该能提供下面这些价值：

- 更稳定的跨会话连续性
- 更好的任务续接能力
- 更稳地保留用户偏好和约束
- 更少依赖模型临场决定是否调用工具
- 更清晰地区分推理职责和上下文管理职责
- 更容易观测和排查记忆系统行为

## 10. 什么算成功

第一版成功的标准是：

- agent 能在对的时刻恢复关键上下文
- agent 能在更少重复解释的情况下继续任务
- 记忆检索是定向的，而不是宽泛且噪声很多
- 注入后的记忆确实提升回复质量，但不会压爆上下文
- 写回生成的是有用记忆，而不是对话垃圾堆积

## 11. 当前工作定义

当前对这个产品的工作定义是：

这是一个由三个协同部分组成的 agent 记忆产品：
一部分负责记忆的检索与写入，
一部分负责在关键时刻把相关上下文主动恢复到 agent 中，
一部分负责把记忆内容和运行过程可视化、可观测地展示出来。

在实现上，这三部分必须保持独立边界，不能做成一个无法拆分的大模块。

## 12. 下一步

在这份基线之后，下一步设计任务是：

定义三部分产品的具体契约，包括：

- 记忆记录结构
- 记忆包结构
- 召回触发规则
- 注入接口
- 写回策略
- 观测事件和页面展示结构
