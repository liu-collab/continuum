# 首版收口开发方案

## 1. 文档目的

这份文档用于把当前仓库从“方向正确的产品原型”收成“边界清楚、可继续开发和验收的首版方案”。

这次收口只解决三类阻塞问题：

1. 工作区隔离与全局记忆边界
2. 治理闭环
3. 端到端契约一致性

同时把前面分析里已经暴露出来的几个主要问题一起收进来，避免继续出现“文档一套、接口一套、页面一套”的情况。

## 2. 当前判断

当前仓库已经有真实主链路：

- `storage` 有写回接收、异步处理、冲突和读模型投影
- `retrieval-runtime` 有触发、检索、注入、写回和观测
- `visualization` 有目录页、轨迹页、看板页

其中和 `storage` 当前阶段直接相关的闭环已经收齐：

- `scope=user` / `scope=workspace` 边界已经按正式口径落地
- `confirm / invalidate / delete / edit / archive / restore_version` 已经形成存储侧治理闭环
- `GET /v1/storage/records` 和 `POST /v1/storage/write-back-candidates` 已经和正式契约对齐
- 冲突治理已经具备新旧双方双边可治理结构

问题不在“有没有代码”，而在“能不能作为一个可信产品继续收口”。

当前真正卡住上线准备度的，不是性能，而是下面三件事：

- 工作区隔离和全局记忆边界还没有真正收清楚
- 错误记忆没有形成用户可控的治理闭环
- 文档、接口、页面和测试没有统一在一套正式契约下

## 3. 当前主要问题

### 3.1 工作区隔离与全局记忆边界没有收清楚

当前产品不是多租户 SaaS，也不是多人共享后台。

按当前交付方式，用户是把 `npm package`（npm 安装包）装到自己本地，默认只有一个本地用户。

所以当前真正要解决的问题不是“多用户权限系统”，而是下面两件事：

- 工作区之间怎么隔离
- 全局记忆和工作区记忆怎么区分并被用户选择

当前实现里，`workspace`（工作区级）虽然在文档和数据模型里存在，但没有被稳定定义成正式产品能力。

直接后果是：

- 文档一会儿把 `workspace` 写成正式范围，一会儿又写成默认关闭
- runtime 还没有把“全局记忆 + 工作区记忆”的选择机制收成正式主链路
- 页面和接口还没有统一反映“当前这条记忆是全局的还是工作区的”
- 用户虽然确实只有一个，但切换项目或目录时，记忆边界还不够清楚

这会直接影响下面三件事：

- 当前轮到底该读哪类记忆
- 写回到底该落成全局记忆还是工作区记忆
- 用户切换项目后，会不会把不该带过去的记忆继续带过去

### 3.2 治理闭环收口前的问题

收口前，`storage` 已经有部分治理能力，比如：

- 编辑
- 归档
- 冲突处理
- 版本恢复

但还缺“用户真的能纠错”的闭环。

当时的主要缺口是：

- 没有正式的 `confirm / invalidate / delete`（确认 / 失效 / 删除）动作口径
- `visualization` 现在还是只读查看，没有治理入口
- `retrieval-runtime` 虽然默认只查 `active`（有效）记录，但没有和用户修正动作形成闭环
- 错误记忆被发现后，还缺一条清楚的“修正后不再继续放大”的产品路径

当前状态补充：

- `storage` 侧治理动作和审计已经闭环
- 剩余待收内容主要在页面治理入口和跨服务联调展示

### 3.3 契约一致性不足

现在最明显的问题不是接口崩，而是“看起来支持，实际上不成立”。

已经看到的典型问题有：

- 运行轨迹页暴露了 `thread_id`、`workspace_id`、`task_id` 筛选，但 runtime 观测接口正式支持的参数并不一致
- 记忆详情页展示 `Created`（创建时间），但共享读模型当前没有稳定发布 `created_at`
- 看板里一部分“当前窗口 vs 上一窗口”比较，实际只是重复请求同一个当前指标接口
- 运行轨迹里展示的依赖状态是全局最新状态，不是该 trace 当时的历史快照

这类问题的共同点是：

- 用户会以为产品支持某个能力
- 页面会展示一个看起来合理的字段
- 但后端并没有按同一契约保证它

## 4. 收口原则

这次收口固定下面四条原则：

### 4.1 先收边界，再收完整

先把“全局记忆 / 工作区记忆边界、治理、页面口径”收好，再谈更多能力。

### 4.2 首版先收成 `single_local_user`（单本地用户）

当前正式交付先固定成：

- 单用户
- 本地安装
- 本地宿主接入

在这个前提下，当前首版正式支持两类长期记忆边界：

- `user`：全局记忆
- `workspace`：工作区记忆

这里的 `user` 不再代表“多用户系统中的某个用户租户”，而是“当前本地安装实例下的全局记忆空间”。

### 4.3 不支持的能力不能假装支持

如果后端没有正式支持：

- 页面不展示这个筛选项
- 文档不把它写成正式范围
- 测试不把它当已交付能力

### 4.4 契约、页面、测试三者必须同口径

一条能力只有在下面三件事同时成立时，才算正式进入产品范围：

- 契约写清楚
- 页面或宿主真实按契约消费
- 自动化测试验证这个契约真的成立

## 5. 当前正式边界

这次收口后的首版正式范围先固定成下面这样：

### 5.1 正式交付模式

- 默认模式是 `single_local_user`
- 当前本地安装实例只有一个用户
- 工作区由宿主传入或由 `cwd`（当前工作目录）映射得到
- 记忆正式分成“全局记忆”和“工作区记忆”

### 5.2 正式开放的作用范围

当前正式开放：

- `session`
- `task`
- `workspace`
- `user`

其中：

- `user` 表示当前本地实例的全局记忆
- `workspace` 表示当前工作区记忆

### 5.3 正式治理动作

首版必须正式支持下面这些治理动作：

- `confirm`（确认）
- `edit`（修正）
- `invalidate`（标记失效）
- `archive`（归档）
- `delete`（软删除）
- `restore_version`（恢复历史版本）

### 5.4 正式运行轨迹筛选范围

当前正式支持的运行轨迹筛选只保留：

- `turn_id`
- `session_id`
- `trace_id`
- `page`
- `page_size`

`thread_id`、`workspace_id`、`task_id` 不再在首版里假装是正式筛选项。

## 6. 解决方案一：工作区隔离与全局记忆边界

### 6.1 固定首版记忆边界模型

当前首版正式采用下面这套边界模型：

- `user`：全局记忆
- `workspace`：工作区记忆
- `task`：当前任务链路记忆
- `session`：当前会话临时记忆

这里的重点不是权限校验，而是使用边界：

- 全局记忆会跨工作区复用
- 工作区记忆只在当前工作区复用
- 任务和会话记忆继续保持短期边界

### 6.2 固定 runtime 默认召回策略

runtime 要把“读哪类记忆”收成正式规则，而不是临时实现。

当前首版固定为：

- `session_start`：默认读 `user + workspace`
- `task_start` / `task_switch` / `before_plan`：默认读 `workspace + task + user`
- `before_response`：按当前 phase 决定读 `workspace + task + user + session`

同时补一条选择规则：

- 用户可以选择当前运行模式是“启用全局记忆 + 工作区记忆”
- 也可以选择“只启用工作区记忆”

首版不要求做复杂策略，先收成一个明确开关：

- `memory_mode=workspace_only`
- `memory_mode=workspace_plus_global`

### 6.3 固定写回策略

写回时要明确区分“这条记忆该进全局，还是该进工作区”。

当前首版固定下面规则：

- 用户偏好、长期表达方式、通用工作习惯：优先写成 `user`（全局记忆）
- 项目约束、仓库约定、工作区规则、项目背景：写成 `workspace`
- 当前任务状态：写成 `task`
- 本轮短期上下文：写成 `session`

如果写回引擎不能稳定判断，就优先保守一点：

- 先落 `workspace`
- 不要把不确定的项目信息写成全局记忆

### 6.4 各服务改动

#### `retrieval-runtime`

- 正式支持 `workspace` scope 召回
- 补一个明确的 `memory_mode`
- 把 phase 到 scope 的映射改成包含 `workspace`
- 在写回候选生成里明确区分全局记忆和工作区记忆

#### `storage`

- 正式支持 `workspace` scope 写入和治理
- 记录层面要稳定保留 `workspace_id`
- 治理页和列表页要能区分 `user` / `workspace`

#### `visualization`

- 页面上要明确展示一条记忆是“全局记忆”还是“工作区记忆”
- 列表和详情页都要支持按 `scope=user|workspace` 解释
- 运行轨迹页要能解释本轮实际用了哪些 scope

#### `continuum-cli` / 宿主适配器

- 启动时提供默认 `memory_mode`
- 能把当前工作区标识稳定传给 runtime
- 如果用户没有显式指定，默认使用 `workspace_plus_global`

### 6.4 验收标准

- 用户可以明确选择“只用工作区记忆”或“工作区 + 全局记忆”
- 工作区 A 的记忆在“只用工作区记忆”模式下不会跑到工作区 B
- 全局记忆在“工作区 + 全局记忆”模式下可以跨工作区复用
- 页面能明确看出一条记忆属于全局还是工作区

## 7. 解决方案二：治理闭环

### 7.1 先收最小可用闭环

首版先不做复杂运营后台，先把下面这条链路收完整：

1. 发现错误记忆
2. 在页面里看到来源和状态
3. 执行确认 / 修正 / 失效 / 删除
4. 存储层写审计和版本
5. 共享读模型更新
6. `retrieval-runtime` 后续不再继续召回错误版本

当前补充：

- `storage` 已经在这个最小闭环基础上继续收成双边冲突治理闭环
- 冲突新候选和旧记录都会落成正式可治理对象
- 冲突单会稳定记录 `pending_record_id` 和 `existing_record_id`

### 7.2 正式状态语义

这次收口固定下面几种状态语义：

- `active`：可以进入 runtime 默认召回
- `pending_confirmation`：待确认，不进入默认召回
- `superseded`：已被更新，不进入默认召回
- `archived`：归档，不进入默认召回
- `deleted`：软删除，只保留审计，不进入默认召回

### 7.3 存储层要补齐的治理动作

当前正式治理动作固定为：

- `confirm`
- `edit`
- `invalidate`
- `archive`
- `delete`
- `restore_version`

同时要求：

- 每个动作都写治理审计
- 每个动作都能追到操作者和原因
- 每个动作都触发共享读模型刷新

### 7.4 页面侧要补齐的治理入口

`visualization` 不再只是只读目录。

首版至少补下面两个入口：

- 记忆详情页：执行确认 / 失效 / 归档 / 删除 / 恢复版本
- 冲突查看页或冲突面板：查看冲突、选择生效版本、记录处理说明

不要求首版做复杂批量治理，但单条治理必须可走通。

### 7.5 运行时联动

`retrieval-runtime` 和治理动作之间要形成固定联动：

- 默认只查询 `active`
- `pending_confirmation` 不进入默认注入
- `superseded`、`archived`、`deleted` 永不进入默认注入
- 当记录被治理后，后续查询不能继续命中旧状态

### 7.6 验收标准

- 用户可以在页面上看到一条记忆的来源、状态和治理历史
- 对一条错误记忆执行 `invalidate` 或 `delete` 后，后续运行时不再默认召回它
- 冲突记忆可以被人工确认并刷新结果
- 治理动作在存储层、共享读模型和运行时结果里是一致的

## 8. 解决方案三：契约一致性

### 8.1 以 `docs/api-contract.md` 为正式对外契约

从这次收口开始，正式口径固定为：

- `docs/api-contract.md` 负责服务接口与共享读模型
- `docs/memory-module-contract.md` 负责运行模块之间的对象与边界
- 页面和宿主如果没有后端正式契约支撑，就不能把能力暴露出来

### 8.2 先删掉“假能力”

当前阶段先做减法：

- 运行轨迹页先只保留后端真正支持的筛选项
- 没有正式发布的字段不在页面上展示成“看起来已有”
- 没有真实时间窗对比数据时，不再把同一快照假装成前后窗口对比

### 8.3 补齐几个关键缺口

这次收口顺带补下面几个关键缺口：

#### 运行轨迹依赖状态

当前轨迹页显示的依赖状态要改成“该 trace 当时的快照”，不能继续复用全局最新状态。

#### 共享读模型字段

记忆详情页如果继续展示 `Created`，那 `created_at` 就要进入共享读模型正式发布字段。

如果短期不准备发布这个字段，就把页面展示拿掉。

这里不允许再保留“页面有字段，但数据模型没有正式字段”的状态。

当前补充：

- `storage` 共享读模型已经正式发布 `created_at`
- `source.origin_workspace_id` 也已经进入正式发布字段口径

#### 看板趋势

趋势比较必须来自：

- 窗口化的原始轨迹 / 作业数据
- 或者明确的窗口化指标接口

不能通过重复请求同一个当前指标接口来伪造“上一窗口”。

### 8.4 自动化契约检查

这次收口要求补三类测试：

#### 契约快照测试

- runtime 观测接口
- storage 观测接口
- 共享读模型字段

#### 页面到接口一致性测试

- 页面暴露的筛选项必须都能被后端消费
- 页面使用的字段必须来自正式契约

#### 端到端验收测试

- 宿主接入 -> runtime -> storage
- storage -> 共享读模型 -> visualization
- 治理动作 -> 运行时结果变化

### 8.5 验收标准

- 页面不再暴露后端未正式支持的筛选项
- 契约文档、接口返回和前端消费结构一致
- 依赖快照、共享字段、看板趋势都不再用“近似值冒充正式值”

## 9. 实施顺序

### 9.1 第一阶段：工作区隔离与全局记忆边界

目标：

- 正式模式收成 `single_local_user`
- 正式支持“全局记忆 + 工作区记忆”边界

交付物：

- `workspace` scope 召回与写回正式落地
- `memory_mode` 开关
- phase 到 scope 的正式映射

### 9.2 第二阶段：治理闭环

目标：

- 从“只读查看”变成“可纠错”

交付物：

- `confirm / invalidate / delete` 治理接口
- 治理审计与版本查看
- 页面治理入口

### 9.3 第三阶段：契约一致性

目标：

- 文档、接口、页面和测试统一

交付物：

- 收窄后的正式筛选项和字段
- 契约快照测试
- 端到端验收测试

### 9.4 第四阶段：回归验收

目标：

- 三件事一起成立

验收重点：

- 工作区边界有效
- 错误记忆可纠正
- 页面不再展示伪能力

## 10. 当前不纳入首版收口的内容

这次不放进当前正式范围：

- 多用户多工作区共享记忆网络
- 复杂运营后台
- 跨组织授权模型
- 自适应策略学习
