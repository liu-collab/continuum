# 可视化与观测平台产品文档

## 1. 文档目的

这份文档只讨论第三部分：`可视化与观测平台`。

它不是记忆检索模块的一部分，也不是记忆注入模块的一部分。
它是独立于前两部分之外的统一观测入口。

一句话说，这个平台负责把“记忆库里有什么”和“系统这一轮做了什么”清楚展示出来。

## 2. 产品定义

可视化与观测平台是整个记忆产品的观察层和排查层。

它的职责是：

- 展示结构化记忆内容
- 展示召回、注入、写回过程
- 展示核心运行指标
- 提供排查入口和解释入口

它不参与记忆检索决策，也不参与记忆注入决策。

它负责的是：

- 看见
- 解释
- 排查

而不是：

- 决策
- 推理
- 执行

## 3. 要解决的问题

如果没有独立的可视化与观测平台，后面很容易出现几个问题：

- 记忆库里到底存了什么，看不清
- 为什么这一轮查了某些记忆，看不清
- 为什么没有注入或没有写回，看不清
- 系统到底是查得慢、写得慢，还是策略错了，看不清

前两部分能运行，不代表产品可用。
要真正能调、能排、能运营，就必须有第三部分。

## 4. 产品目标

可视化与观测平台的首版目标是：

- 让结构化记忆内容可查看
- 让关键运行过程可追踪
- 让核心指标可查看
- 让问题排查有统一入口

对应的成功标准是：

- 可以快速看到当前记忆库的内容结构
- 可以解释一轮运行里为什么查了、为什么注入了、为什么写回了
- 可以用指标定位问题大概出在检索、注入还是写回

## 5. 产品边界

这个平台负责：

- 展示数据
- 聚合事件
- 呈现指标
- 提供筛选和查看能力

这个平台不负责：

- 改写记忆内容
- 直接干预 agent 运行
- 代替前两部分做业务决策
- 承担底层存储本身

## 6. 输入来源

这个平台的输入来自前两部分的结构化输出。

### 6.1 来自记忆检索与写入模块

至少包括：

- 结构化记忆记录
- 记录状态变化
- 查询事件
- 写入事件
- 合并、拒绝、归档等处理结果
- 查询和写入耗时

### 6.2 来自记忆注入模块

至少包括：

- 触发事件
- 召回判断结果
- 实际注入结果
- 未注入原因
- 写回检查结果
- 写回候选生成结果

## 7. 页面结构

首版建议拆成三类核心页面。

### 7.1 记忆库页面

用于看“当前记忆库里有什么”。

至少展示：

- 记忆列表
- 记忆类型
- 作用范围
- 状态
- 摘要
- 来源
- 最近更新时间

这个页面重点是结构化查看，不是展示原始聊天流水。

### 7.2 运行轨迹页面

用于看“一轮交互里发生了什么”。

至少展示：

- 是否触发召回
- 触发原因
- 查了哪类记忆
- 返回了多少结果
- 最终注入了什么
- 是否形成写回候选
- 写回候选的处理结果

这个页面重点是解释链路。

### 7.3 指标看板页面

用于看“系统整体运行得怎么样”。

至少展示：

- 记忆写入接受率
- 记忆合并率
- 重复记录率
- 空检索率
- 查询耗时
- 写入耗时
- 召回命中率
- 触发后实际注入率
- 写回候选通过率

这个页面重点是发现趋势和定位问题。

## 8. 首版交互要求

首版先满足最基本的查看和排查能力：

- 可以筛选记忆类型
- 可以筛选作用范围
- 可以按任务或会话查看运行记录
- 可以查看某条记忆的来源和状态变化
- 可以查看某一轮的召回到写回全过程

首版不要求复杂编辑能力。

## 9. 关键指标

第三部分至少要统一展示下面这些指标：

- 记忆写入接受率
- 记忆合并率
- 重复记录率
- 空检索率
- 查询耗时
- 写入耗时
- 召回命中率
- 不必要召回率
- 触发后实际注入率
- 写回候选通过率

这些指标的目标不是“好看”，而是“能用来判断哪里有问题”。

## 10. 技术要求基线

### 10.1 数据接入方式

第三部分通过 REST 接口从前两部分获取可观测数据。

主要接入点：

- `/memory/observe`：统一的可观测数据输出接口，由前两部分提供
- 第三部分定期拉取或通过事件推送接收运行事件
- 必要时也可以读取 `storage` 发布的共享读模型中的只读视图，用于记忆列表和状态展示

接入约束：

- 只消费前两部分正式输出的结构化数据，不直接读取运行时私有状态或数据库。
- 如果读取数据库，只能读取 `storage` 发布的共享读模型，不能读取私有写表。
- 数据格式必须和前两部分在契约层约定好，不能依赖对方内部实现细节。
- 接入链路不能影响前两部分的主链路性能。

### 10.1.1 数据源不可用时的行为

如果 `storage` 或 `retrieval-runtime` 未启动或不可达：

- `visualization` 自身仍然正常启动
- 页面仍然可访问
- 对应区域显示“数据源不可用”或等价状态
- 不因为上游服务没启动就让页面服务本身不可运行

### 10.2 性能指标基线

首版固定下面这组 P95 目标：

| 指标 | P95 目标 | 说明 |
|------|---------|------|
| 可观测事件写入 | ≤ 1s | 非关键路径，不阻塞前两部分 |
| 页面数据查询 | ≤ 2s | 面向平台用户，不在 agent 主链路上 |
| 指标聚合刷新 | ≤ 30s | 看板数据允许短暂延迟 |

这部分的性能要求整体低于前两部分，因为它不在 agent 主链路上。但页面响应仍然要保证基本可用性。

## 11. 首版范围

首版先做这些：

- 结构化记忆列表页
- 单轮运行轨迹页
- 核心指标看板
- 基础筛选能力
- 面向前两部分的统一观测数据接入
- 通过 REST 接口接入可观测数据
- 性能基线目标

不做这些：

- 复杂 BI 分析
- 自定义报表系统
- 直接在线编辑记忆
- 复杂权限系统

## 12. 与前两部分的关系

第三部分和前两部分的关系要固定成下面这样：

- 它消费前两部分的结构化输出
- 它不反向控制前两部分的核心行为
- 它是独立产品，不是前两部分里的附属页面

这点很重要。

因为如果把页面挂在检索模块里，页面就会被理解成“存储后台”。
如果把页面挂在注入模块里，页面又会被理解成“运行调试页”。

但你这里想要的，其实是一个统一的、可观测的产品视角。

## 13. 一句话定义

可视化与观测平台是记忆产品的第三部分。

它独立于记忆检索与写入、记忆注入之外，负责把记忆内容、运行过程和关键指标统一展示出来。
