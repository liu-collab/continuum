# 产品补全层

这份文档补的是当前产品骨架之外，还必须补上的三层：

- 场景边界
- 治理闭环
- 验收机制

另外，产品实现还必须满足一条硬性架构约束：

- 三部分代码必须可独立拆出、独立治理、独立运行

这里的落地口径进一步固定为：

- 三部分是独立服务
- 允许共享由 `storage` 发布的只读数据库读模型
- 不允许共享写模型和私有实现
- 任意一个服务未启动或不可达，都不能影响其他服务自身的运行状态

这条约束的详细定义单独见 `architecture-independence.md`。

前面的文档已经把 `存储`、`检索/注入`、`可视化` 三部分能力讲清楚了。
这份文档解决的是：这套能力到底服务谁、怎么持续可信地运行、以及上线前怎么算真的可用。

## 1. 场景边界

场景边界的目标，是防止产品做着做着变成一个什么都想管的“泛记忆平台”。

### 1.1 目标用户

首版先明确两类用户：

- 直接用户：和 agent 交互的终端用户
- 平台用户：配置、观察、排查这套记忆系统的产品或运营人员

这里先不把“多组织、多团队协同运营”放进首版主场景。

### 1.2 核心使用场景

首版先只覆盖下面几类核心场景：

- 跨会话续接任务
- 记住稳定偏好和约束
- 记住当前任务状态
- 在关键节点恢复历史决策
- 让用户可以查看记忆和系统行为

首版先不覆盖下面这些场景：

- 强人格化陪伴
- 复杂长期心理画像
- 多 agent 共用一个复杂记忆网络
- 跨产品、跨工作区的大范围共享记忆

### 1.3 作用范围边界

记忆必须明确作用范围，首版固定下面四层：

- `session`（会话级）：只在当前会话有效
- `task`（任务级）：只在当前任务链路有效
- `user`（用户级）：跟随用户跨会话复用
- `workspace`（工作区级）：团队或项目共享的公共约束

默认规则：

- 会话级最短
- 用户级最稳定
- 工作区级必须比用户级更严格

### 1.4 可见性边界

不是所有记忆都能被所有请求看到。

首版先固定下面几条：

- 用户级记忆默认仅当前用户可见
- 任务级记忆默认仅当前任务链路可见
- 工作区级记忆必须显式标记为可共享
- 任何跨范围读取都必须有明确授权规则

### 1.5 隔离边界

为了避免后面出事故，首版要先定清楚隔离规则：

- 不同用户之间的记忆必须隔离
- 不同工作区之间的记忆必须隔离
- 测试环境和正式环境必须隔离
- 调试页面默认不能跨用户直接浏览原始记忆

### 1.6 来源边界

只有满足条件的内容，才允许进入记忆系统：

- 来自用户明确表达
- 来自 agent 已完成的任务状态更新
- 来自用户或系统明确确认的重要事件

默认不把下面这些内容直接当作可信记忆来源：

- 纯模型猜测
- 未确认的隐含推断
- 中间推理草稿

### 1.7 首版一句话边界

首版产品边界可以收成一句话：

这是一个面向单用户任务续接和偏好保持的 agent 记忆产品，支持清晰范围隔离、结构化存储、运行时召回和可观测查看。

## 2. 治理闭环

治理闭环的目标，是让记忆系统长期可用，而不是写进去以后越积越乱。

### 2.1 为什么必须有治理闭环

如果没有治理闭环，后面一定会出现：

- 记忆写错了但改不掉
- 旧信息和新信息冲突
- 低价值记忆不断堆积
- 用户不信任系统记住的内容

所以产品不能只有“写入成功”，还要有“写错以后怎么办”。

### 2.2 治理动作

首版先明确下面几类治理动作：

- 编辑
- 删除
- 归档
- 标记失效
- 标记冲突
- 人工确认
- 恢复历史版本

这些动作不一定全部开放给终端用户，但产品上必须预留。

### 2.3 责任分层

治理动作需要分层：

- 自动治理：去重、合并、归档、失效判断
- 半自动治理：检测到冲突后给出建议
- 人工治理：对关键事实、共享记忆、冲突记忆做人工确认或修正

### 2.4 纠错闭环

首版至少要有下面这条纠错链路：

1. 发现错误记忆
2. 标记问题来源
3. 修正、合并或失效处理
4. 更新状态和索引
5. 在后续检索中不再继续错误放大

如果这条链路没有，系统就会把错误越用越牢。

### 2.5 生命周期规则

记忆需要生命周期，而不是永久活跃。

首版先固定：

- 会话级记忆可自动过期
- 任务级记忆在任务结束后归档或摘要化
- 用户级稳定偏好长期保留，但允许覆盖更新
- 工作区级公共规则必须有版本和生效状态

### 2.6 冲突治理

冲突是记忆产品一定会碰到的核心问题。

首版先定下面规则：

- 冲突记忆不能静默覆盖
- 冲突来源必须可追溯
- 高可信新信息可以覆盖低可信旧信息
- 无法自动解决时，降级为待确认状态

### 2.7 用户可控性

如果用户不能理解也不能纠正记忆，产品信任感会很差。

首版至少要支持：

- 用户知道系统记住了什么
- 用户知道某条记忆来自哪里
- 用户可以要求删除或修正明显错误的记忆

### 2.8 安全与合规底线

这部分先作为产品底线写清楚：

- 敏感信息默认不入长期记忆
- 共享范围必须显式控制
- 关键治理动作要保留审计记录
- 删除和失效处理需要可追踪

### 2.9 首版一句话闭环

首版治理闭环可以收成一句话：

系统可以自动整理记忆，但对关键事实、冲突和共享内容，必须保留人工确认与修正通道。

## 3. 验收机制

验收机制的目标，是防止产品“看起来功能都有”，但实际上不可上线。

### 3.1 验收对象

首版要验收的不只是单点功能，而是三类能力一起验收：

- 存储是否稳定
- 检索/注入是否有效
- 可视化是否能解释系统行为

### 3.2 MVP 验收标准

首版 MVP 至少满足下面这些标准：

- 用户稳定偏好可以跨会话保留
- 任务状态可以在下次进入时续接
- 检索延迟满足运行时要求
- 写入不会明显拖慢检索
- 页面能看到结构化记忆和关键运行指标
- 错误记忆可被发现、标记和修正
- 任意一个服务未启动时，其他服务仍能独立启动并返回正常或降级结果

### 3.3 指标验收

首版至少建立下面几类验收指标：

- 用户重复说明率
- 任务续接成功率
- 空检索率
- 噪声检索率
- 查询耗时
- 写入耗时
- 写回接受率
- 错误记忆率
- 触发后实际注入率

### 3.4 评测集

首版不能只靠主观感觉验收，至少要准备一套场景评测集。

评测集建议包含：

- 偏好记忆场景
- 任务续接场景
- 任务切换场景
- 冲突记忆场景
- 错误写回场景
- 空记忆场景

每类场景都要有预期结果，方便回归比较。

### 3.5 接入验收

如果它要成为一个独立产品，还需要验收“外部 agent 能不能稳定接上来”。

首版至少要验收下面这些内容：

- 外部 agent 能否按约定发起查询
- 外部 agent 能否按约定提交写回候选
- 外部 agent 能否消费统一记忆包
- 平台能否正确记录运行事件并在页面展示

### 3.6 上线闸门

首版上线前，建议至少过下面几道闸门：

- 功能闸门：核心链路全部可用
- 性能闸门：查询速度达标
- 数据闸门：没有明显错误隔离和越权问题
- 观测闸门：页面可查看关键过程和关键指标
- 治理闸门：错误记忆有处理路径
- 可用性闸门：任意单服务未启动时，其他服务不被拖挂

### 3.7 发布策略

首版不建议一上来全量。

更稳的发布方式是：

- 先内部试用
- 再小范围灰度
- 最后扩大到正式场景

每一阶段都看同一套核心指标。

### 3.8 首版一句话验收

首版验收可以收成一句话：

这套产品只有在“能记住、能取回、能解释、能纠错、能评估”五件事同时成立时，才算真正可上线。
