## 一、核对范围

本次判断主要对照以下内容：

- `services/memory-native-agent/docs/roadmap-v0.2.md`
- `services/memory-native-agent/docs/tiered-memory-injection-plan.md`
- `services/memory-native-agent/src/runner/conversation.ts`

- `services/memory-native-agent/src/http/routes/health.ts`
- `services/memory-native-agent/src/providers/shared.ts`

判断口径如下：

- `合理性`：这项能力对 MNA 后续扩展是否必要
- `真实性`：对当前仓库现状的描述是否准确
- `优先级`：按当前已落地程度和风险重新排序，不直接沿用“理想形态”下的排序

## 二、快速结论

| 项目 | 原判断 | 当前仓库实际情况 | 结论 |
| --- | --- | --- | --- |
| 上下文窗口管理 | 说成“基本没有” | 已有 token 估算、历史裁剪、归档摘要、工具输出截断 | 方向对，但表述偏重 |
| 规划与任务分解 | 说成“没有规划能力” | 有 `before_plan` 阶段触发与任务状态，但没有真正计划器 | 基本成立 |
| 自我评估与纠错 | 说成“失败只原样回给 LLM” | provider 有基础重试，agent 层无质量验收与策略切换 | 部分成立 |
| 工具执行并行化 | 说成“工具调用串行” | 同轮多个 `tool_call` 仍逐个 `await` | 成立 |
| 沙箱 | 说成“只有 deny pattern，直接跑宿主机” | 有确认、超时、终止，但无隔离、快照、回滚 | 基本成立 |
| 缓存层 | 说成“没有缓存” | 只有零散的会话级授权复用，不是通用缓存层 | 基本成立 |
| 可观测性 | 说成“只有基础计数器” | 已有事件流、`trace_id`、JSON metrics，但没有标准 trace/export | 方向对，但表述偏重 |

一句话总结：

`这 7 项里，2/4/5/6 基本属实，3 是部分属实，1 和 7 被说重了。`

## 三、逐项判断

## 1. 上下文窗口管理

### 当前已经有的能力

- `Conversation.buildMessages()` 已不再是简单全量拼接，而是经过 `compactMessages()`
- `token-budget.ts` 已提供基于字符估算的 token 预算控制
- 已有 `context.max_tokens`、`context.reserve_tokens`、`context.compaction_strategy`
- 历史消息超长时会做截断，或在 `summarize` 模式下生成历史摘要
- `conversation.ts` 还有 `archivedSummary`，会把较早消息压成摘要块
- 工具输出已支持内联截断和落 `artifact`
- `fs_read` 已支持 `max_lines` / `byte_limit`
- `shell_exec` 已支持 `max_output_bytes`

### 还缺的部分

- token 估算仍是粗粒度字符估算，不是 provider 级精确 tokenizer
- system prompt、记忆注入、工具结果、历史消息之间还没有显式预算配比
- 工具输出截断和会话 token budget 仍是两层逻辑，尚未统一联动
- 没有基于语义价值的历史压缩策略，目前仍偏规则式裁剪

### 判断

- `合理性`：高
- `真实性`：部分成立

更准确的说法应是：

`MNA 已有基础上下文预算机制，但距离生产级的上下文编排引擎还有明显差距。`

---

## 2. 规划与任务分解

### 当前已经有的能力

- `trigger-detector.ts` 能识别 `task_start`、`task_switch`、`before_plan`
- `AgentRunner` 会记录 `currentTask` 和 `recentTasks`
- `before_plan` 已进入记忆检索阶段，可为“规划前注入”提供上下文
- system prompt 里有“先简述方案，再写代码”的提示

### 当前没有的能力

- 没有单独的计划对象，也没有稳定的数据结构承载计划
- 没有“执行前给用户展示计划并确认”的机制
- 没有步骤状态机，不知道当前在第几步
- 没有计划修订逻辑，也没有计划执行历史
- 当前的 `before_plan` 更像“规划前准备上下文”，不是“规划器”

### 判断

- `合理性`：高
- `真实性`：成立

更准确的说法应是：

`当前 MNA 只有任务触发和阶段标记，还没有真正的计划生成、跟踪和修订能力。`

---

## 3. 自我评估与纠错

### 当前已经有的能力

- provider 层有基础重试策略，覆盖超时、流错误、限流、部分不可用场景
- 工具调用有错误结构化返回，不是完全裸异常
- 权限拒绝、超时、模式拦截等错误已分类型处理

### 当前没有的能力

- agent 层没有“结果验收”步骤，执行后不会判断是否真的达成用户目标
- 没有基于错误类型切换执行策略的统一框架
- 没有连续失败后的反思或降级策略
- 没有 turn 级的最大重试次数和防循环保护
- 工具失败后通常只是把结果继续放回上下文，缺少系统侧纠偏

### 判断

- `合理性`：高
- `真实性`：部分成立

更准确的说法应是：

`MNA 不是完全没有重试，而是只有 provider / tool 级的基础错误处理，没有 agent 级自我评估与纠错闭环。`

---

## 4. 工具执行并行化

### 当前情况

`agent-runner.ts` 里，同一轮如果 provider 连续返回多个 `tool_call`，当前逻辑仍是：

- 收到一个 `tool_call`
- 立即 `await runTool(...)`
- 写回 `tool` 消息
- 再继续处理下一个

这意味着：

- 多个独立读操作无法并发
- 多个搜索 / 多文件读取场景会被串行拉长
- 当前 turn loop 还没有“同轮收集后并发执行”的结构

### 判断

- `合理性`：高
- `真实性`：成立

这是当前 7 项里最明确、争议最小的一项缺口。

---

## 5. 沙箱

### 当前已经有的能力

- `shell_exec` 有用户确认
- 有 `deny_patterns`
- 有超时控制和进程终止
- 文件工具会校验 workspace 边界
- 工具调用有审计记录

### 当前没有的能力

- `shell_exec` 仍直接在宿主机执行
- 没有文件系统级隔离
- 没有网络级隔离
- 没有工作区快照
- 没有回滚机制
- 没有真正的资源配额隔离

### 判断

- `合理性`：高
- `真实性`：基本成立

更准确的说法应是：

`MNA 现在有的是“权限门控 + 边界校验 + 超时终止”，不是“真正的沙箱执行环境”。`

---

## 6. 缓存层

### 当前已经有的能力

- `PermissionGate` 对 `allow_session` 有会话级复用
- 部分运行态有内存态缓冲，例如会话事件缓冲

### 当前没有的能力

- 没有通用的工具结果缓存
- 没有 prompt / response 级缓存
- 没有 retrieval / embedding 结果缓存
- 没有会话上下文缓存抽象
- 也没有统一的 TTL / LRU / 失效策略

### 判断

- `合理性`：中到高
- `真实性`：基本成立

这里需要补一句：

`当前不是绝对零缓存，但现有缓存还停留在局部小优化，不能称为“缓存层”。`

---

## 7. 可观测性

### 当前已经有的能力

- `http/state.ts` 里有运行态 metrics 聚合
- `/v1/agent/metrics` 已提供 JSON 指标输出
- 事件流里已有 `phase_result`、`tool_call_start`、`tool_call_result`、`turn_end`
- 多处链路都保留了 `trace_id`
- `dispatched_messages` 也会保存 prompt 片段和 phase 结果，便于调试回放

### 当前没有的能力

- 没有标准 span 结构
- 没有完整层级 trace
- 没有 OTLP 导出
- 没有 Prometheus 文本格式导出
- 没有统一的错误归因视图
- 延迟统计仍基于内存采样和手工 percentile

### 判断

- `合理性`：高
- `真实性`：部分成立

更准确的说法应是：

`MNA 已有基础观测面板和运行指标，但还没有进入标准化 tracing / metrics export 的阶段。`

---

## 四、修正后的优先级建议

如果按“当前仓库实际缺口”重新排序，更合适的顺序如下。

### P0 — 不补齐就难以上到更稳定的生产场景

1. `上下文窗口管理深化`
   - 不是从零开始重做
   - 而是把现有 token budget 提升成真正的上下文编排
2. `沙箱`
   - 当前确认和 deny pattern 只能降低风险
   - 还不能提供隔离与回滚能力

### P1 — 不补齐就很难稳定处理复杂任务

1. `规划与任务分解`
2. `自我评估与纠错`
3. `工具执行并行化`

说明：

- `工具并行化` 很重要，但它更偏吞吐和执行效率问题
- 从“能不能安全上线”来看，它比沙箱靠后
- 从“复杂任务会不会跑偏”来看，它也不如规划和纠错关键

### P2 — 不补齐会限制运维和成本优化

1. `可观测性标准导出`
2. `缓存层`

说明：

- 这两项都很适合在核心执行链更稳定之后再抽象
- 其中可观测性往往先做导出，缓存则应在热点路径清晰后再做

---

## 五、最终判断

对这 7 项需求，当前更稳妥的结论可以写成：

1. `上下文窗口管理` 不是空白，而是已经有基础版，接下来需要做“深化”，不是“补零”
2. `规划与任务分解` 当前确实还没正式落地
3. `自我评估与纠错` 目前只有基础错误处理，没有 agent 级闭环
4. `工具执行并行化` 是真实缺口，而且实现边界相对清晰
5. `沙箱` 当前仍停留在门控层，不是隔离执行环境
6. `缓存层` 当前只有零散局部缓存，没有形成正式能力
7. `可观测性` 已有基础设施，但离标准 tracing 和外部监控集成还差一层

最终可归纳为：

`这份 7 项清单整体方向是对的，但不应该把 MNA 现状描述成“几乎全空白”。更准确的说法是：MNA 已经补上了一部分基础能力，下一阶段要做的是把这些基础能力升级成可扩展、可生产、可观测的正式模块。`
